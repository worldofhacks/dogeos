# Architecture & design

Part of the [2026-06-11 multi-agent repo audit](../README.md). Every finding below was independently challenged by an adversarial verifier agent that re-read the cited code; verdicts and calibrated severities are recorded inline.

**Result: 8 confirmed, 0 refuted.**

## Strengths

- Clean acyclic dependency DAG with correct layering: config and dogeos-rpc are leaves; aggregator depends only on them; api composes aggregator; the web server wraps api; the React app talks only HTTP. A full import-graph sweep of packages/aggregator/src found zero cycles — sources/registry, fees, routes, quotes, swap, verification form a strict hierarchy.
- Uniform dependency injection as the backbone of testability: createAggregatorApiHandler (handler.mjs:379-408) takes every side effect (clock, verifiers, quote providers, calldata builder, activity provider) as an injectable parameter, and live.mjs is a textbook composition root that wires real RPC implementations. Result verified: 250 tests pass in ~1.4s with no network.
- Frontend layering discipline unusual for a solo project: lib/execute.js is a framework-agnostic swap lifecycle (approval → swap → receipt polling) with zero React/DOM imports, consumed by a thin useSwapExecution hook; quote math lives in lib/quote.js; views are almost pure presentation.
- The aggregator core (quoteService.mjs, 216 lines) is small, pure, and BigInt-precise, with an explicit rejection taxonomy (wrong-chain / source-excluded / not-active) and deterministic tie-breaking — the most money-sensitive code in the JS stack is the cleanest.
- The aggregator does expose a real curated facade (index.mjs, 86 lines of named re-exports) with a dedicated publicApi.test.mjs guarding it — the gap is only that live.mjs bypasses it, not that it is missing.
- Defensive details in infrastructure code: path-traversal guard in the static server (packages/web/src/server.mjs:99-111), XSS-hardened runtime-config injection (replaceAll('<','\\u003c'), server.mjs:49), and in-flight quote request coalescing to protect the RPC (handler.mjs:409-439).
- Code comments are honest engineering records rather than noise: SwapView documents exactly which UI numbers are real vs placeholder ("HONESTY: no USD sublabels (no price feed)"), and useWallet.js documents the prior connect-flow bug and why the current design exists.

## Confirmed findings

### 1. Chain identity and network constants are hardcoded independently in the frontend, duplicating packages/config

**CONFIRMED** · Severity: **Medium** · Kind: fact

- **Where:** packages/config/src/chains.mjs:2-3; apps/web/src/lib/api.js:4,10; apps/web/src/sdkConfig.js:5; apps/web/src/sdk-chain-switch.js:9,14; apps/web/src/injected-wallet.js:1-9; apps/web/src/lib/execute.js:19,39-41
- **What:** packages/config/src/chains.mjs:2-3 defines `id: 6_281_971, idHex: "0x5fdaf3"` as the single source of truth, but the frontend never imports it (Vite could bundle it — `grep packages/` in apps/web/src returns nothing). Instead the chain id is re-hardcoded in apps/web/src/lib/api.js:4 (`export const DOGEOS_CHAIN_ID = 6_281_971`), sdkConfig.js:5, sdk-chain-switch.js:9 and 14 (also embeds the RPC URL in an error string), and injected-wallet.js:1. The faucet URL is defined twice within the frontend itself: lib/api.js:10 and lib/execute.js:19.
- **Why it matters:** The mainnet cutover this project is explicitly heading toward requires synchronized edits in at least six files; missing one leaves a wallet chain-guard (`chainIdMatchesDogeos`, execute.js:39-41) pinned to the testnet id, which would block every transaction — or worse, leave a switch path pointing wallets at the wrong chain. This is the highest-leverage consolidation available before real funds are involved.
- **Adversarial verifier:** Confirmed in full. Every cited line is exact: chains.mjs defines id 6_281_971/0x5fdaf3 as the nominal source of truth, yet grep shows zero imports of packages/config anywhere in apps/web/src — the chain id is independently hardcoded in api.js:4, sdkConfig.js:5, sdk-chain-switch.js:9 (with the RPC URL and id additionally baked into an error string at line 14), and injected-wallet.js:1 (which also freezes full wallet_addEthereumChain params at lines 3-9), and the faucet URL is duplicated between api.js:10 and execute.js:19. The "Vite could bundle it" premise is proven by vite.config.mjs itself importing from packages/api and packages/web. The consequence is accurate: execute.js:39-41 (chainIdMatchesDogeos) gates every approval/swap send on the hardcoded id, so a mainnet cutover missing any of the 6+ files either fail-closes the app or leaves switch/add-chain paths pointing wallets at the stale testnet chain/RPC. No mitigation exists: KNOWN_ISSUES.md and README do not acknowledge the duplication, and packages/config/test/dogeosConfig.test.mjs only pins the config package's own values — there is no cross-file consistency test. Medium is calibrated: no present fund-loss (dominant failure mode is the fail-closed chain guard) but a genuine, unmitigated hazard sitting directly on the planned testnet-to-mainnet path.

### 2. Wallet state crosses two React roots via a mutable window global and stringly-typed events, with event names duplicated in three files

**CONFIRMED** · Severity: **Medium** · Kind: judgment

- **Where:** apps/web/src/sdk-wallet.jsx:8-9,41-47,78-82; apps/web/src/index.html:15; apps/web/src/sdk-wallet-provider.jsx:29-30,124-159; apps/web/src/ui/useWallet.js:47-55,62-67 (plus prior-bug comment at 26-29); apps/web/src/lib/execute.js:22-24,153-160; apps/web/src/ui/useTokenBalances.js:18-21; apps/web/src/main.jsx:15 (fourth raw event literal)
- **What:** The wallet bridge is a second React root (`createRoot` on `#sdk-wallet-root`, sdk-wallet.jsx:78; mount point in index.html) that publishes state to the main app by assigning `window.dogeosAggregatorWallet = bridge` (sdk-wallet.jsx:41, sdk-wallet-provider.jsx:141) and dispatching CustomEvents. The event-name constants `"dogeos:sdk-wallet-updated"` / `"dogeos:sdk-wallet-ready"` are re-declared in sdk-wallet.jsx:8-9, sdk-wallet-provider.jsx:29-30, and useWallet.js:47-49; the global is read raw in useWallet.js:53-55, useTokenBalances.js:19, and lib/execute.js:22-24. Readiness is a 10s timeout race (useWallet.js:62-67).
- **Why it matters:** The contract between the wallet subsystem and the swap-execution path exists only as matching string literals and a window global — a renamed event or a load-order change fails silently at runtime (the exact class of bug the long comment at useWallet.js:26-29 says already happened once). The two-root design is a deliberate lazy-load tradeoff, but the implicit shape of the bridge object (openModal/getProvider/switchToDogeOS/...) is duplicated knowledge in five consumer files with no single definition.
- **Adversarial verifier:** Every factual element verifies: two React roots (createRoot on #sdk-wallet-root at sdk-wallet.jsx:81, mount in index.html:15), the mutable window.dogeosAggregatorWallet global assigned in both sdk-wallet.jsx:41 and sdk-wallet-provider.jsx:141, the event-name string pair re-declared verbatim in three files (sdk-wallet.jsx:8-9, sdk-wallet-provider.jsx:29-30, useWallet.js:47-48 — and a third event "dogeos:load-sdk-wallet" duplicated raw in main.jsx:15, slightly worse than claimed), raw global reads in useWallet.js:53-55, useTokenBalances.js:19, execute.js:22-24, and the 10s readiness timeout at useWallet.js:62-67. The implicit bridge shape (openModal/getProvider/getChainId/switchToDogeOS/...) is produced in two files and consumed via optional chaining in three more with no shared definition, so a rename degrades silently. Mitigation is weak: staticApp.test.mjs:60,77 only regex-checks two of the strings in two producer files — no consumer files are tested, no cross-file consistency assertion exists, and there is no JS CI at all (only the contracts workflow). No acceptance in KNOWN_ISSUES.md or README. The useWallet.js:26-29 comment does document a prior silent contract failure in this bridge, though it was an unhandled-rejection/preference bug rather than literally an event rename — the "exact class" phrasing is a mild stretch, not a refutation. Severity Medium is calibrated: no fund-loss path (worst case is silently frozen wallet state or a 10s-delayed connect error breaking the swap UI), but it sits on the swap-execution critical path with zero behavioral coverage and documented precedent of this failure class.

### 3. handler.mjs mixes HTTP dispatch with financial policy and an outbound integration

**CONFIRMED** · Severity: **Medium** · Kind: judgment

- **Where:** packages/api/src/handler.mjs:80-99, 252-266, 292-335, 341-377, 123-138, 409-439, 448-710
- **What:** The 712-line handler is well injected and tested, but it embeds non-HTTP concerns: the Blockscout activity client (`fetchBlockscoutAddressTransactions`, handler.mjs:80-99) is an external integration living in the route file; `inferredSwapSlippageBps` (252-266) reverse-engineers slippage from quote deltas (`ceilDiv((maxAmountIn - amountIn) * 10_000n, amountIn)`) — financial policy; `refreshSwapQuote` (292-335) re-runs the full quote pipeline; `mergeVenueVerification` (341-377) joins registry and verification domain data; plus a request-coalescing cache keyed on `JSON.stringify` (123-138, 409-439). Routing itself is a linear if-chain over `url.pathname` (448-710).
- **Why it matters:** The slippage-inference and pre-build re-quote rules are the most safety-relevant logic in the API yet they live outside the aggregator package where the routing/fee math and its tests are concentrated; they cannot be reused by scripts or future consumers without the HTTP layer. The file is at the size where every new endpoint compounds the if-chain and (per the duplication finding) requires edits in two other files.
- **Adversarial verifier:** Every citation is exact: the Blockscout client (80-99), inferredSwapSlippageBps with the verbatim ceilDiv expression (252-266, line 258), refreshSwapQuote re-running buildQuoteResponse (292-335), mergeVenueVerification (341-377), the JSON.stringify-keyed coalescing cache (123-138, 409-439), and the linear pathname if-chain (444/448-710). The reuse consequence is verified: createAggregatorApiHandler is the module's ONLY export, so the slippage-inference and pre-build re-quote logic are module-private and unreachable from scripts/, while the forward slippage math lives separately in packages/aggregator/src/quoteService.mjs:19-24 (floor division) with no shared code enforcing the round-trip with the handler's ceil-division inverse. The consequence is amplified by facts the auditor partly cited: live.mjs:157 defaults refreshSwapQuoteBeforeBuild=true (production path the frontend depends on per apps/web/src/lib/execute.js:14), packages/web/src/server.mjs:12-21 hardcodes an API_PATHS allowlist and apps/web/src/lib/api.js duplicates routes (three-file edits per endpoint), and docs/superpowers/specs/2026-06-07-dogeos-routing-engine-spec.md section B4 plans to land Permit2/router-fallback/revert-mapping logic in this same file. Mitigations exist and cap severity: full dependency injection, 1402-line handler.test.mjs covering refreshSwapQuoteBeforeBuild and slippage propagation plus 766-line live.test.mjs, the web client always sends explicit slippageBps (execute.js:257) making inference a fallback that rounds up conservatively, and on-chain minOut/maxIn enforcement means no fund-loss path. No documented acceptance in KNOWN_ISSUES.md/README. As a judgment finding with accurate facts and a plausible, partially-verified consequence, it is real; Medium is calibrated for a pre-mainnet project whose spec routes more safety-critical work into this file.

### 4. API route list and Node-to-Fetch HTTP bridge are hand-duplicated across three server stacks

**CONFIRMED** · Severity: **Low** (auditor proposed Medium, calibrated by verifier) · Kind: fact

- **Where:** packages/api/src/server.mjs:7-50; packages/web/src/server.mjs:11-22,54-97; vite.config.mjs:6-17,27-70; packages/api/src/handler.mjs:448-653 (e.g. "/quote" at 547)
- **What:** Three independent HTTP entry points each carry a verbatim copy of the IncomingMessage→Request bridge (`hasBody`, `headersFromIncomingMessage`, `readIncomingBody`, `requestFromIncomingMessage`, `writeFetchResponse`): packages/api/src/server.mjs:7-50, packages/web/src/server.mjs:54-97, vite.config.mjs:27-70. On top of that, the set of API paths is re-declared as a literal `API_PATHS = new Set(["/sources", "/tokens", ... "/swap"])` in packages/web/src/server.mjs:11-22 and again in vite.config.mjs:6-17, while the authoritative routes live as inline string comparisons in handler.mjs (e.g. `url.pathname === "/quote"` at handler.mjs:547).
- **Why it matters:** Adding or renaming an endpoint in handler.mjs works under `start:api` but silently 404s (falls through to static file serving) under the production web server and the Vite dev server until two more files are edited; any bug fix in the request-bridging code (chunked bodies, header arrays) must be applied three times or the servers diverge.
- **Adversarial verifier:** Fully confirmed as fact: the IncomingMessage-to-Request bridge (hasBody/headersFromIncomingMessage/readIncomingBody/requestFromIncomingMessage/writeFetchResponse) is hand-copied in all three cited files with no shared module, and the copies have already diverged slightly (api server.mjs:14 iterates message.headers without the `?? {}` guard present in the other two copies, plus naming drift), which directly evidences the claimed divergence risk. API_PATHS is literally duplicated in packages/web/src/server.mjs:11-22 and vite.config.mjs:6-17 while handler.mjs defines routes only as inline pathname string comparisons (line 547 verified) and exports no route list; the web server (server.mjs:157) and Vite middleware (vite.config.mjs:90) gate on their local sets, so a new handler route does 404/fall through there until two more edits — the claimed consequence is accurate. No mitigations exist: no parity test ties the sets to handler routes (web tests only round-trip /quote), and nothing in README or audit/KNOWN_ISSUES.md documents acceptance. Severity downgraded from Medium to Low: this is a maintainability/duplication issue with no security or fund-loss path, and the failure mode (404 on a newly added endpoint) is loud and immediately discoverable during normal dev because the Vite dev server requires the same edit, so it would rarely ship unnoticed.

### 5. No package boundary mechanism — single root package.json, and boundary erosion is already visible

**CONFIRMED** · Severity: **Low** (auditor proposed Medium, calibrated by verifier) · Kind: fact

- **Where:** package.json (root, name "dogeswap", no workspaces); packages/api/src/live.mjs:2-17; packages/aggregator/src/index.mjs:22-86; packages/web/test/injectedWallet.test.mjs:4-9; packages/web/src/server.mjs:9-10
- **What:** There is exactly one package.json (`name: "dogeswap"`, no `workspaces` field; verified via `find . -name package.json`), so `packages/*` are directories, not packages, and nothing enforces the boundaries the layout implies. Erosion has already happened: packages/api/src/live.mjs:1-17 imports 12 deep internal paths (`"../../aggregator/src/discovery/v2Pools.mjs"`, `"../../aggregator/src/swap/calldataRegistry.mjs"`, etc.) even though every one of those names is re-exported by the facade packages/aggregator/src/index.mjs (lines 22, 45, 51-77) which handler.mjs correctly uses; packages/web/test/injectedWallet.test.mjs imports `"../../../apps/web/src/injected-wallet.js"`; packages/web/src/server.mjs:9-10 hardcodes `new URL("../../../apps/web/src/")` / `dist/` paths into a sibling app.
- **Why it matters:** The aggregator facade (which has a dedicated publicApi.test.mjs) does not actually protect internal structure: any file move inside packages/aggregator/src breaks packages/api, and any directory rename breaks the `../../..` path math in four packages at once. For a solo founder heading to mainnet this is the main thing that makes refactoring the routing core risky; the fix is cheap now (npm workspaces or even just routing all cross-package imports through each index.mjs) and expensive later.
- **Adversarial verifier:** Every cited fact verifies: the repo has exactly one project-owned package.json (root, no workspaces field; other hits are vendored forge libs/charting library), so packages/* are plain directories with no boundary enforcement. live.mjs really does deep-import 12 aggregator internals (lines 2-17) even though all 12 names are re-exported by the facade index.mjs that handler.mjs correctly uses — I checked each name individually. The injectedWallet test and web server.mjs ../../.. cross-package paths are also as cited (plus more of the same in sdkChainSwitch.test.mjs, server.test.mjs, staticApp.test.mjs). publicApi.test.mjs only asserts facade exports and cannot stop consumers bypassing it. No documented acceptance in README or KNOWN_ISSUES, and verify-repository-scope.mjs guards EVM-contract scope, not imports. The finding is real, but severity is downgraded from Medium to Low: the failure mode is loud ERR_MODULE_NOT_FOUND at startup/test time, and the root test script globs all packages (node --test "packages/**/*.test.mjs", 250 tests, ~1.3s), so any file move breaking live.mjs or the path math is caught instantly during refactoring. This is maintainability friction with zero runtime, security, or fund-loss exposure on a pre-deploy testnet project — accurate, cheap to fix (workspaces or facade-only imports), but housekeeping rather than risk.

### 6. Quote diagnostics and per-request block pinning rely on hidden mutation of the request input object

**CONFIRMED** · Severity: **Low** (auditor proposed Medium, calibrated by verifier) · Kind: fact

- **Where:** packages/api/src/handler.mjs:109-117,553 and packages/api/src/live.mjs:53-68,112-132 (supporting: handler.mjs:411-439, packages/aggregator/src/routes/oneHop.mjs:91-101)
- **What:** `attachQuoteDiagnostics` installs a non-enumerable `quoteDiagnostics` array on the parsed input via `Object.defineProperty(input, "quoteDiagnostics", { enumerable: false ... })` (handler.mjs:109-117); providers report errors by pushing into it (`input.quoteDiagnostics.push(diagnostic)`, live.mjs:112-115). Separately, `createRequestBlockNumberProvider` pins all quotes in one request to a single block by keying a WeakMap on the input object's identity (live.mjs:53-68).
- **Why it matters:** Two load-bearing contracts — error telemetry and same-block quote consistency — are invisible in every function signature and depend on object identity plus a hidden property. Any natural refactor that clones, freezes, or reconstructs the input (the code already serializes it with JSON.stringify for cache keys, which silently drops the hidden field) would break block pinning without any error, yielding candidates quoted at different blocks and lost source-error telemetry.
- **Adversarial verifier:** All cited facts verified verbatim: attachQuoteDiagnostics installs a non-enumerable quoteDiagnostics array via Object.defineProperty (handler.mjs:109-117, attached to parsed.input at :553), providers push into input.quoteDiagnostics with a silent early-return guard (live.mjs:112-115), and createRequestBlockNumberProvider pins blocks via a WeakMap keyed on input identity (live.mjs:53-68). The fragility is not hypothetical — it already manifests in-repo: createOneHopQuoteCandidateProvider spreads the input per leg ({...input, buyToken: viaToken}, oneHop.mjs:91-101); spread copies only enumerable props, so leg source/provider errors are silently dropped from telemetry, and the new object identity misses the WeakMap so one-hop legs quote at unpinned, freshly-fetched blocks (composeOneHopCandidates even reconciles differing leg blocks via earlierBlockNumber) — exactly the predicted zero-error breakage, though one-hop candidates are status "readOnly" previews, never executable. However, the claim overstates "would break without any error": both contracts are pinned by fast unit tests — live.test.mjs:409-411 asserts exactly one eth_blockNumber call with a uniform block across quoter invocations, live.test.mjs:442-450 asserts sourceErrors telemetry, and handler.test.mjs:542-609/763-804 assert the diagnostics contract including deliberate replay of cloned sourceErrors for coalesced in-flight requests (handler.mjs:417, showing the authors handled the JSON.stringify cache-key path the claim flags). A clone/freeze refactor of the direct path would fail node --test, though there is no JS CI to enforce it. No fund-loss path exists: /swap re-quotes via refreshSwapQuote (which attaches diagnostics to its own fresh input, handler.mjs:300-301) and on-chain settlement enforces minimum-output in DogeSwapRouter. Downgraded Medium→Low: real hidden-coupling/maintainability finding with a demonstrated silent gap, but impact is confined to telemetry completeness and preview quote consistency, with test guardrails covering the executable path.

### 7. Production web server silently falls back to serving raw JSX source when the build is missing

**CONFIRMED** · Severity: **Low** · Kind: fact

- **Where:** packages/web/src/server.mjs:9-10,33-35; apps/web/src/index.html:18
- **What:** `defaultStaticRoot()` returns `DEFAULT_SOURCE_ROOT` (apps/web/src) whenever `apps/web/dist/index.html` is absent (packages/web/src/server.mjs:9-10, 33-35). That index.html loads `<script type="module" src="/main.jsx">` — raw JSX the browser cannot parse. The fallback is a leftover from the removed vanilla app.js era (the file would have been plain JS then; the vanilla app was deleted in commit 513945d).
- **Why it matters:** If the build step is ever skipped (the Dockerfile and `start:web` script do chain `vite build`, so this is an edge path), the server responds 200 with uncompilable source and a blank page instead of failing loudly — a confusing failure mode that costs debugging time and serves no purpose anymore; it should hard-fail when dist is missing.
- **Adversarial verifier:** Confirmed: defaultStaticRoot() (server.mjs:33-35) silently falls back to apps/web/src when apps/web/dist/index.html is absent, and that index.html loads <script type="module" src="/main.jsx"> (index.html:18). The consequence is actually worse than claimed: CONTENT_TYPES (server.mjs:25-31) has no .jsx entry, so /main.jsx is served as application/octet-stream and browsers hard-block non-JS-MIME module scripts — a guaranteed blank page behind a 200. Both blessed run paths build first (start:web chains vite build; Dockerfile runs build:web at image build), so it is an edge path as the auditor said. One narrative correction: the fallback is not a functional vanilla-era leftover — at the commit introducing server.mjs (82c4c9f) index.html already referenced /main.jsx (app.js was already orphaned), so the fallback never served a working app. Partial mitigation: packages/web/test/server.test.mjs intentionally exercises source-root serving (and a no-staticRoot test relies on the fallback to pass on unbuilt checkouts), but tests pin staticRoot explicitly so a startup hard-fail in startAggregatorWebServer would remain test-compatible. No README/KNOWN_ISSUES acceptance. Low severity is correctly calibrated: pure ops/debugging footgun, no fund-loss or security impact (path traversal is guarded at server.mjs:102-107).
