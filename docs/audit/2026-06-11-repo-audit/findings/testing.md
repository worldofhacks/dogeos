# Testing

Part of the [2026-06-11 multi-agent repo audit](../README.md). Every finding below was independently challenged by an adversarial verifier agent that re-read the cited code; verdicts and calibrated severities are recorded inline.

**Result: 7 confirmed, 0 refuted.**

## Strengths

- Calldata tests assert exact byte-for-byte ABI encoding: venueCalldataBuilders.test.mjs builds the expected hex word-by-word and asserts full calldata equality including the 4-byte selector for all six venue paths (e.g. `assert.equal(builder.selector, "0x38ed1739")` then full-string compare, packages/aggregator/test/venueCalldataBuilders.test.mjs:64-68) — the strongest possible assertion for the bytes users sign.
- Route-selection and slippage math are pinned to exact BigInt values, not approximations: quoteService.test.mjs asserts `minimumOutput === 1_044_750n` for 50bps on 1_050_000n, exact net-output scoring, gas-aware ranking for both exact-input and exact-output modes, and a full rejection taxonomy (wrong-chain/stale/not-active/source-excluded) (packages/aggregator/test/quoteService.test.mjs:47-58,103-110,188-222).
- handler.test.mjs covers genuinely hard concurrency behavior most projects never test: in-flight quote coalescing, candidate-work reuse across slippage-only and source-ordering changes, quote refresh before /approval and /swap, simulation gas attachment, and balance-preflight refusal (packages/api/test/handler.test.mjs:542,611,684,1004,1314).
- Contracts testing maps 1:1 to documented invariants: RouterInvariants.t.sol implements I1-I8 exactly as INVARIANTS.md claims, using a Trail-of-Bits-style ghost-variable handler, plus an Echidna assertion harness for stranded-fund drain and a cleanly vm.skip-gated fork differential test; all 53 forge tests pass locally including the live fork test (packages/contracts/test/RouterInvariants.t.sol:74-131, test/echidna/EchidnaRouter.sol, test/fork/RouterFork.t.sol:44-47).
- The mocked RPC harnesses assert the JSON-RPC call sequence itself (e.g. `assert.deepEqual(rpc.calls.map((call) => call.method), ["eth_chainId", "eth_gasPrice"])`, packages/api/test/server.test.mjs:113-116), so tests catch redundant or missing chain round-trips, not just response handling.
- Wallet-bridge edge cases get real behavioral coverage: 21 tests for EIP-6963 discovery including adversarial impersonation (Rainbow mimicking MetaMask metadata, MyDoge-as-window.ethereum, no auto-select with multiple wallets, add-chain fallback flows) (packages/web/test/injectedWallet.test.mjs:101-584).
- The suite is fast and deterministic by construction — injected `nowMs` everywhere, fixture timestamps, zero network — 250 tests in 1.28s, which makes the missing CI integration a one-line fix rather than an infrastructure project.

## Confirmed findings

### 1. Entire React UI money path (apps/web hooks, components, lib/units.js, lib/execute.js) has zero tests and is excluded from the test glob

**CONFIRMED** · Severity: **Medium** (auditor proposed High, calibrated by verifier) · Kind: fact

- **Where:** package.json:7; apps/web/src/lib/units.js:1-34 (decimalToUnits/sanitizeAmountInput); apps/web/src/lib/execute.js:1-17; apps/web/src/lib/quote.js:1-31; packages/web/test/injectedWallet.test.mjs:9; packages/web/test/sdkChainSwitch.test.mjs:7; packages/api/src/handler.mjs:148,206 (backend validates amount positivity only)
- **What:** Root test script is `"test": "node --test \"packages/**/*.test.mjs\" \"scripts/**/*.test.mjs\""` — apps/** is not in the glob and `find apps -name '*.test.*'` returns nothing. Untested: SwapView.jsx (990 lines), SwapFlow.jsx (702), all five hooks (useQuote/useSwapExecution/useTokenBalances/useWallet/useSettings), and the deliberately extracted pure modules: lib/units.js (`decimalToUnits` — "1.25", 18 -> base units via `BigInt(whole) * 10n ** BigInt(decimals) + BigInt(paddedFraction)`), lib/execute.js (approval→swap→receipt wallet tx lifecycle, `hexQuantity`, chain/account guards), lib/quote.js, lib/api.js. These files self-describe as "No DOM, no React — pure functions" extracted "so React components and hooks can share the exact same on-chain math" — i.e. designed for testability — yet only injected-wallet.js and sdk-chain-switch.js are imported by packages/web/test.
- **Why it matters:** decimalToUnits/sanitizeAmountInput are the direct path from user keystrokes to the on-chain amount the user signs. A rounding/padding/decimals bug there produces a wrong-but-valid amount that passes every backend check (backend only validates > 0) and the contract's minOut (which is derived from the same wrong amount). On a real-money mainnet deploy this is the most likely silent fund-loss vector, and it is the one layer with no safety net.
- **Adversarial verifier:** Every cited fact verified: the root test glob covers only packages/** and scripts/** (package.json:7), no *.test.* exists anywhere under apps/, and of the 5 files in packages/web/test only injected-wallet.js and sdk-chain-switch.js are behaviorally imported from apps/web — staticApp.test.mjs and server.test.mjs just regex over source text. The 250-test suite passes without touching units.js, execute.js, quote.js, api.js, or any hook/component, despite units.js/execute.js self-describing as pure modules extracted for shared on-chain math. The consequence chain also holds: the API validates amountIn only via toPositiveBigInt (>0), balancePreflight only catches over-balance amounts, the backend re-quote reuses the client-supplied amount, and minOut derives from the same wrong amount — so a within-balance decimal-conversion regression passes every layer silently. Partial mitigations: the current decimalToUnits/sanitizeAmountInput implementations look correct on inspection (regex guard, single-dot collapse, pad/truncate), and the UI you-receive readout could surface gross errors (though it uses unitsToDecimal from the same untested file). No documented acceptance anywhere (README:67 actually overstates web coverage), and there is no JS CI at all (only contracts-security.yml). Severity downgraded from High to Medium: this is a coverage/process gap with no demonstrated defect on a testnet-only pre-deploy project — but it is the one real keystroke-to-signed-amount fund-loss path with zero safety net, so it should be treated as a hard blocker before any mainnet deploy.

### 4. Foundry invariant fuzzing explores only one fixed 3-command V3 program; native/V2/Algebra/multi-venue settlement paths are never fuzzed

### 7. Wall-clock 100ms watchdog races in provider-composition tests are a latent flake source

**CONFIRMED** · Severity: **Low** · Kind: fact

- **Where:** packages/aggregator/test/v2PoolDiscovery.test.mjs:448; packages/aggregator/test/concentratedLiquidityProvider.test.mjs:182; packages/aggregator/test/quoteProviderComposition.test.mjs:97; packages/api/test/live.test.mjs:213,434
- **What:** Four tests race the code under test against a real timer, e.g. `setTimeout(() => reject(new Error("v2 provider hung")), 100)` in v2PoolDiscovery.test.mjs:448, with the same pattern at concentratedLiquidityProvider.test.mjs:182, quoteProviderComposition.test.mjs:97, and live.test.mjs:434; live.test.mjs:213 additionally sleeps 5ms to order concurrent chain-verification calls.
- **Why it matters:** On a loaded CI runner (once JS CI exists) a >100ms scheduling stall makes these tests false-fail; everything else in the suite is fully deterministic (injected nowMs, mocked fetch), so these are the only timing-sensitive spots and are trivially fixable with longer/promise-based watchdogs.
- **Adversarial verifier:** All five citations are exact and the supporting facts check out: four tests race the code under test against a real 100ms setTimeout watchdog, live.test.mjs:213 sleeps 5ms to keep chain verification in-flight, and a grep across all JS test trees (aggregator, api, web app, scripts) confirms these are the only wall-clock-sensitive spots — everything else uses injected nowMs and mocked fetch. No mitigation exists (no fake timers, no retries, no JS CI yet, nothing in KNOWN_ISSUES). However, the claimed mechanism is overstated: the inner 5ms timeouts are also real setTimeout timers (v2Pools.mjs:139, concentratedLiquidity.mjs:50, composite.mjs:34), and Node fires expired timers in due-time order, so a generic >100ms event-loop stall after both timers are scheduled cannot make the watchdog win — the 5ms timer fires first and the provider settles via microtasks before the watchdog callback runs. A false failure requires a process freeze landing in the microtask-sized gap between scheduling the watchdog and scheduling the inner 5ms timer (the gap exists because the providers await a mocked block-number/chain-verification call first), which is possible under heavy CI preemption/GC but rare; live.test.mjs:213 cannot realistically false-fail at all since stalls only widen the in-flight window it depends on. Net: a genuine but weak latent flake source, trivially fixable (clear watchdog on settle, longer or due-ordered-safe watchdogs), with zero fund-loss or product impact and no JS CI to flake yet — Low is correctly calibrated, at the very bottom of Low.
