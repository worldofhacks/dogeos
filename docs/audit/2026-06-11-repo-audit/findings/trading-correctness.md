# Trading/quoting correctness (core money paths)

Part of the [2026-06-11 multi-agent repo audit](../README.md). Every finding below was independently challenged by an adversarial verifier agent that re-read the cited code; verdicts and calibrated severities are recorded inline.

**Result: 7 confirmed, 0 refuted.**

## Strengths

- No unlimited ERC-20 approvals anywhere: approvals are planned for the exact swap amount only, after reading the live allowance (packages/aggregator/src/swap/erc20Approval.mjs:58-83, packages/api/src/handler.mjs:638-645) — unusually disciplined for a DEX frontend.
- Reference-correct, overflow-free BigInt AMM math: V2 getAmountOut/getAmountIn replicate Uniswap exactly including the +1 ceil on exact-output input and the amountOut>=reserveOut liquidity guard (packages/aggregator/src/quotes/adapters/v2.mjs:19-34); all amount math is BigInt end-to-end with JS Number explicitly confined to display (apps/web/src/lib/units.js:49-58).
- Defense-in-depth before any signature: every /swap is eth_call-simulated AND gas-estimated with a 1.2x ceil-rounded buffer, then balance-preflighted for both the sell token and native gas including the DogeOS L1 data/finality fee (packages/aggregator/src/swap/verifySwapTx.mjs:59-77, packages/aggregator/src/swap/balancePreflight.mjs:56-67, wired by default in packages/api/src/live.mjs:261-275).
- Execution is gated behind a verified-source registry: calldata builders are matched per source+quoteMode, require ACTIVE status, executable ABI provenance, execution verification, and a router-address + output-selector match (packages/aggregator/src/swap/calldataRegistry.mjs:39-84); one-hop composites are correctly marked readOnly and can never reach buildSwapTx (packages/aggregator/src/routes/oneHop.mjs:44).
- Venue fault isolation done well: per-source and per-provider timeouts with Promise.race, errors converted to structured per-request diagnostics surfaced in /quote telemetry instead of failing the whole quote (packages/aggregator/src/discovery/v2Pools.mjs:132-152, packages/aggregator/src/quotes/providers/composite.mjs:27-52, packages/api/src/handler.mjs:593-605).
- Quote staleness enforced at three layers: TTL rejection in route selection (packages/aggregator/src/routes/direct.mjs:5), TTL re-check at tx build (packages/aggregator/src/swap/buildSwapTx.mjs:53), and a UI expiry lock that replaces the confirm button with 'quote expired · refresh' (apps/web/src/ui/SwapFlow.jsx:255-264).
- Strict RPC hygiene: every JSON-RPC result is shape-validated, JSON-RPC errors are raised not swallowed, batch responses are matched by id with missing-entry detection (packages/dogeos-rpc/src/jsonRpcClient.mjs:83-137), and chain id is verified before quoting and swapping (packages/api/src/live.mjs:23-45).
- Fast deterministic test suite (250 tests, ~1.6s) with dedicated coverage of adapters, calldata builders, approval planning, balance preflight, and the quote service — a solid regression net for the math layer.

## Confirmed findings

### 1. /swap silently re-quotes and rebases minAmountOut to the refresh-time price, so the on-chain slippage floor can be arbitrarily below the user-displayed 'min received'

**CONFIRMED** · Severity: **High** · Kind: fact

- **Where:** packages/api/src/handler.mjs:292-335 (refreshSwapQuote, no original-vs-refreshed comparison), 628-637 (/approval), 664-673 (/swap); packages/api/src/live.mjs:157,244 (refreshSwapQuoteBeforeBuild=true live default, used by both servers); packages/aggregator/src/quoteService.mjs:19-21,52-57 (minOut recomputed from fresh amountOut); packages/aggregator/src/swap/buildSwapTx.mjs:53-55 (TTL check on refreshed quote only); apps/web/src/lib/execute.js:309-340 (approval wait then unguarded sign of /swap tx)
- **What:** With refreshSwapQuoteBeforeBuild=true (the live default), /approval and /swap each call refreshSwapQuote(), which rebuilds the route and recomputes minimumOutput = freshAmountOut*(10000-slippageBps)/10000 (quoteService.mjs: `(route.amountOut * (BASIS_POINTS - slippageBps)) / BASIS_POINTS`). buildSwapTx only TTL-checks the REFRESHED quote, never the quote the user saw. The web flow (execute.js executeSwap) sends the approval tx, waits up to 120s for its receipt, then posts /swap and immediately signs the returned transaction (`const swap = await postSwap({ quote, sender }); ... const txHash = await sendWalletTransaction(transaction, sender);`) without comparing swap.quote.amountOut/minAmountOut against the originally displayed route. SwapFlow's expiry lock only gates the confirm click, not the approval→swap window.
- **Why it matters:** Slippage tolerance is enforced relative to the market price at build time, not the price the user accepted. If the price moves 10% down during the approval wait (30-120s is normal), the user receives ~10% + slippage less than the displayed 'min received' with zero warning and no bound. On mainnet with real funds this is the classic stale-quote execution loss; industry practice is to pin minOut to the user-acknowledged quote or re-surface the new quote before signature.
- **Adversarial verifier:** Confirmed end-to-end. The live wiring defaults refreshSwapQuoteBeforeBuild=true (live.mjs:157; both packages/api/src/server.mjs:87 and packages/web/src/server.mjs:147 use createLiveAggregatorApiHandler without override), so /approval and /swap each rebuild the route and recompute minAmountOut = freshAmountOut*(10000-slippageBps)/10000 (quoteService.mjs:19-21,52-57); refreshSwapQuote carries only recipient/deadline/sender from the user's quote (handler.mjs:283-290) and performs no comparison against the originally displayed amountOut — no deviation guard exists anywhere (grep confirmed). buildSwapTx TTL-checks the refreshed quote's fresh timestamp so it always passes. The web client displays min-received from the original /quote route (quote.js:64-68, SwapView.jsx:862) but signs the /swap transaction blind after the up-to-120s approval wait (execute.js:218,309-340); SwapFlow's expiry lock only gates the confirm click (SwapFlow.jsx:613-645). handler.test.mjs:941-1002 asserts the rebasing as intended behavior (user maxAmountIn 1,010,000 silently becomes 1,212,000 — 20% worse — after refresh), proving design intent with no bound. Mitigations are partial only: user deadline (default 300s) bounds time not price, refresh is pinned to the same sourceId/amountIn, native-DOGE sells skip the approval window; nothing bounds how far the enforced floor falls below the displayed value, and the public approval tx pre-signals the swap making the window targetable. No acceptance documented in KNOWN_ISSUES.md or README; only an execute.js comment framing the re-quote as a freshness feature. High (not Critical) is calibrated: real fund-loss path for a mainnet-bound aggregator, but slippage is still applied to a genuine fresh same-venue price, the window is deadline-bounded, and the project is currently testnet-only/pre-deploy.

### 2. Barkswap Algebra quoter result decoding contradicts the project's own 'verified' ABI artifact; failure mode is silent venue death or a wrong exact-output price

**CONFIRMED** · Severity: **High** · Kind: fact

- **Where:** packages/aggregator/src/discovery/concentratedLiquidityPools.mjs:213-227,269 vs packages/aggregator/src/abi/adapterAbiArtifacts.mjs:242-287 (wired as 'verified' evidence at packages/aggregator/src/sources/registry.mjs:298,380-387; silent-failure sink at packages/aggregator/src/quotes/providers/concentratedLiquidity.mjs:104)
- **What:** The live decoder reads 6+ output words: `const quotedFeeTier = decodeWord(result, 5, ...)`, `gasUnits: decodeWord(result, 4, ...)`, `quotedAmountIn: decodeWord(result, 0, ...)`. But BARKSWAP_ALGEBRA_QUOTER_ABI — the artifact used as ADAPTER_ABI_PROVENANCE evidence for this ACTIVE source — declares only 2 outputs: `outputs: [{ name: "amountOut", type: "uint256" }, { name: "fee", type: "uint16" }]` (fee at word 1, no gasEstimate). Both cannot match the deployed contract. The only tests covering this (concentratedLiquidityLiveDiscovery.test.mjs:263-331) use self-made fakeClient fixtures shaped to match the decoder, so they prove nothing about the real quoter. If the artifact shape is what's deployed, decodeWord(result,5) throws on every quote and `.catch(() => null)` (line 269) swallows it with no quoteDiagnostics entry — the venue produces zero quotes while telemetry reports sourceErrorCount 0. If the deployed shape is the 6-word Integral tuple, the artifact 'verification' is wrong and the word-0=amountIn assumption for exactOutput is unvalidated against the chain.
- **Why it matters:** An active execution venue's quote correctness rests on an unresolved internal contradiction. Best case: Barkswap silently never quotes and the aggregator degrades to fewer venues with no alert. Worst case (exactOutput via API): quotedAmountIn actually decodes the amountOut word, producing a wrong price, a wrong maxAmountIn bound, and a wrong ERC-20 approval amount. Needs a one-time live cross-check of the raw quoter return against the decoder before mainnet.
- **Adversarial verifier:** Confirmed, and the worst case is stronger than claimed. The live decoder assumes a >=6-word Algebra quoter return (fee at word 5, gas at word 4, amount at word 0; decodeWord throws on short data) while the project's own BARKSWAP_ALGEBRA_QUOTER_ABI artifact — stamped status:'verified' and attached as quoter evidence for the ACTIVE barkswap-algebra source — declares exactly 2 static outputs (amountOut/amountIn uint256 + fee uint16), i.e. 2 returndata words; both cannot describe the same deployed contract. The only covering tests (concentratedLiquidityLiveDiscovery.test.mjs:263-331) use hand-built 6-word fakeClient fixtures, proving nothing about the chain. The silent-death path is real: quotePool(...).catch(()=>null) at line 269 swallows decode throws, the null output returns [] at concentratedLiquidity.mjs:104 WITHOUT invoking onSourceError, so quoteDiagnostics stays empty and handler.mjs:603 reports sourceErrorCount 0. Live verification (verificationSnapshot.mjs:308-310) only does bytecode selector-presence and factory()/poolDeployer() relationship reads — it never validates quoter output shape — and no KNOWN_ISSUES/README entry documents the contradiction. Additionally, I verified upstream cryptoalgebra/Algebra QuoterV2 (the 'QuoterV2-style' family the registry note itself claims, and the only family consistent with 6 return words) returns (amountOut, amountIn, sqrtPriceAfter, ticksCrossed, gasEstimate, fee) for BOTH directions — so even if the 6-word assumption is right, the exact-output decode of word 0 as quotedAmountIn actually reads amountOut, yielding a nonsense price, corrupted exact-output route ranking, wrong maxAmountIn, and a wrong ERC-20 approval amount; the repo's own test fixture ordering contradicts upstream. Mitigations that bound fund loss: buildSwapTx runs eth_call simulation (verifySwapTx.mjs:61-63) plus balance preflight, and the venue router enforces amountInMaximum on-chain, so garbage bounds largely fail closed (revert) rather than draining funds — this keeps it below Critical. High is calibrated: active venue, provable internal contradiction in 'verified' provenance, zero-telemetry failure mode, residual over-approval exposure to an unaudited external router, and a cheap fix (one-time live raw-returndata cross-check plus artifact/decoder reconciliation) that should gate mainnet.

### 3. User's tx-deadline setting is silently dropped for MuchFi V3 routes (no deadline in calldata)

**CONFIRMED** · Severity: **Medium** · Kind: fact

- **Where:** packages/aggregator/src/swap/venueCalldataBuilders.mjs:105-117 (also registry.mjs:25-29, execute.js:250-262, buildSwapTx.mjs:74-81, SettingsView.jsx:191-192)
- **What:** buildMuchFiV3ExactInputSingleCalldata / buildMuchFiV3ExactOutputSingleCalldata encode (tokenIn, tokenOut, fee, recipient, amount, bound, sqrtPriceLimitX96) with no deadline word — consistent with the SwapRouter02-style ABI in the registry (`exactInputSingle((address,address,uint24,address,uint256,uint256,uint160))`), and no multicall(deadline,...) wrapper is built. The V2 and Algebra builders DO encode `encodeUint(quote.deadline, "deadline")`. execute.js bindExecutionQuote sets deadline = now + user deadlineSeconds for every route, so the UI implies the setting applies universally.
- **Why it matters:** A MuchFi V3 swap stuck in the mempool (underpriced gas, congestion) can execute hours later with only minAmountOut protection — the user's explicit deadline is unenforced and they have no way to know. SwapRouter02-style routers expect multicall(uint256 deadline, bytes[]) to restore this; either wrap the call or surface 'no deadline on this venue' in the UI.
- **Adversarial verifier:** Confirmed real. buildMuchFiV3ExactInputSingle/ExactOutputSingle encode exactly 7 words with no deadline, matching the SwapRouter02-style ABI (selectors 0x04e45aaf/0x5023b4df) in registry.mjs:25-29, while the V2 (lines 94,102) and Algebra (lines 123,130) builders DO encode quote.deadline; no multicall(deadline,...) wrapper exists anywhere in source. execute.js bindExecutionQuote applies the user's deadlineSeconds to every route, SettingsView.jsx:191-192 labels it 'tx deadline — cancel if not confirmed in time', and SwapFlow.jsx:537-538 displays it on the review screen for V3 routes — so the setting is silently unenforced. No mitigation: buildSwapTx sends the tx directly to the venue router (to: quote.router), not through the deadline-enforcing DogeSwapRouter (deploy-pending); routeBinding.deadline is off-chain metadata; verifySwapTx only does pre-sign eth_call/estimateGas; tests assert the 7-word no-deadline layout as expected behavior. Not documented as accepted risk — KNOWN_ISSUES.md #9 covers only coarse timestamps on the contract; in fact docs/adapter-verification.md:91 lists 'No deadline.' as an Automatic Rejection Condition, which the ACTIVE muchfi-v3 source violates, and the runbook table (docs/dogeos-router-verification-runbook.md:106) shows MuchFi V3 safety fields without deadline. Severity Medium is calibrated: loss is bounded by the still-encoded minAmountOut and DogeOS's sequencer model makes long mempool stalls less likely than L1, but the user grants an open-ended stale-execution option at the worst acceptable price, the UI promises behavior it doesn't deliver, and the repo's own verification policy is breached. Fix per the finding: wrap V3 calls in multicall(deadline, [exactInputSingle...]) or surface the missing-deadline caveat per venue.

### 4. All V2 venue quotes assume a hard-coded 30 bps swap fee that is never validated against the venue

**CONFIRMED** · Severity: **Medium** · Kind: judgment

- **Where:** packages/aggregator/src/discovery/v2Pools.mjs:194 (default feeBps=30n; used at 260/296/316); packages/aggregator/src/quotes/adapters/v2.mjs:19-34; packages/aggregator/src/sources/registry.mjs:19-24 (selector 0xd06ca61f listed, no per-source feeBps); packages/api/src/live.mjs:173-180 (no feeBps override)
- **What:** createLiveV2QuoteCandidateProvider defaults `feeBps = 30n` and live.mjs never overrides it; quotes are computed locally from getReserves() via amountOutWithFee/amountInForExactOutput with that constant. The MuchFi V2 router's getAmountsOut (selector 0xd06ca61f) is verified present in bytecode (registry V2_ROUTER_SELECTORS) but never called to cross-check the assumed fee, and no per-source feeBps exists in the registry.
- **Why it matters:** If MuchFi V2 (or any future V2 venue) uses a non-standard fee (25/20/100 bps forks are common), every displayed amountOut and the derived minAmountOut are systematically wrong: fee >30bps inflates quotes so swaps revert on minOut whenever the error exceeds the slippage setting; fee <30bps systematically under-quotes and loses route ranking. A one-time startup assertion comparing local math to getAmountsOut would eliminate the assumption.
- **Adversarial verifier:** All cited facts confirmed: createLiveV2QuoteCandidateProvider hard-codes feeBps=30n, live.mjs never overrides it, the registry's muchfi-v2 entry carries no fee field, and getAmountsOut (0xd06ca61f) appears only in selector lists/ABI artifacts and tests — it is never called, and verification is selector-substring-in-bytecode only, which cannot distinguish a 25/20/100 bps fork (fee-modified forks keep the identical router ABI). No mitigation or documented acceptance exists: docs' onchain validation covers CLMM fee tiers only, KNOWN_ISSUES' fee items concern the DogeSwapRouter protocol fee, and there is no test cross-checking local math against getAmountsOut. The CL adapter takes feeBps from the live quoter output, so the assumption is V2-specific exactly as claimed. Consequence is plausible but bounded — venue-router minAmountOut/maxAmountIn (venueCalldataBuilders.mjs:91-102) plus the eth_call preflight (verifySwapTx.mjs:61) convert a large fee mismatch into blocked/reverting swaps rather than fund loss, and a small mismatch into systematic under-delivery capped by the user's slippage setting — so Medium is correctly calibrated for a trading-correctness finding on an aggregator heading to mainnet (systematic quote error on the only ACTIVE v2 venue, trivially fixable with the proposed one-time getAmountsOut assertion), not High (no unbounded loss) and not Low (every V2 quote and the route ranking depend on an unvalidated constant).

### 5. Route ranking subtracts DOGE-wei gas fees 1:1 from output-token wei (outputWeiPerFeeWei defaults to 1n integer)

**CONFIRMED** · Severity: **Medium** · Kind: fact

- **Where:** packages/aggregator/src/fees/dogeosFeeEstimator.mjs:23-24 (scoring), packages/api/src/live.mjs:143+246 (1n default), packages/aggregator/src/routes/direct.mjs:47-58 (ranking by netOutput), packages/web/src/server.mjs:147 (production path: createLiveAggregatorApiHandler() with no args), packages/api/src/server.mjs:74-94
- **What:** scoreQuote computes `feeCostInOutputToken = fee.totalFeeWei * outputWeiPerFeeWei; netOutput = amountOut - feeCostInOutputToken - failurePenalty`, and the live handler defaults `outputWeiPerFeeWei = 1n` (server.mjs passes nothing). So a fee denominated in DOGE wei is deducted at face value from USDC/WETH/LBTC/USDT output wei regardless of relative token value, and an integer rate cannot express any output token worth more than DOGE per wei (rate < 1). chooseBestDirectRoute ranks by this netOutput.
- **Why it matters:** Venue ranking — the aggregator's core value proposition — is distorted whenever two venues' price gap is smaller than the mis-valued gas delta (e.g. v2 135k vs algebra 180k gasUnits at 1 gwei is ~4.5e13 wei treated as 0.000045 WETH ≈ real-money error of ~1000x). minAmountOut and displayed amounts are unaffected, but users can be routed to a worse-priced venue. Needs a per-pair conversion (even a coarse one) before mainnet.
- **Adversarial verifier:** Confirmed. scoreQuote multiplies a DOGE-wei gas+finality fee by outputWeiPerFeeWei and subtracts it from output-token wei, chooseBestDirectRoute ranks by the result, and every production entrypoint ends at the 1n default (live.mjs:143): the Dockerfile runs packages/web/src/server.mjs which calls createLiveAggregatorApiHandler() with zero args, and api/src/server.mjs passes undefined. Cross-venue gas deltas are real (v2 135k in discovery/v2Pools.mjs:195 vs v3 165k / algebra 180k in quotes/providers/concentratedLiquidity.mjs:11-14), so ranking is biased toward low-gas venues whenever the price gap is below the mis-valued fee delta, overweighting gas ~10^3-10^5x for WETH/LBTC outputs (correct only for WDOGE output). The integer-rate limitation also holds: rates strictly between 0 and 1 (needed for any token more valuable per wei than DOGE) are inexpressible with bigint multiplication. No mitigation found: handler.mjs:302 supports a per-request provider hook (resolveProvider receives the quote input) but nothing wires a real conversion, no env/config knob exists, tests use 1n literally, and neither README, docs/, nor contracts/audit/KNOWN_ISSUES.md documents this as accepted. Two minor softenings, neither changing the verdict: all official testnet tokens are 18 decimals (packages/config/src/tokens.mjs) so there is no 6-decimal magnitude blowup, and minAmountOut/displayed amounts derive from amountOut+slippage (quoteService.mjs:52-57) so there is no fund-loss path — only routing users to a worse-priced venue. Medium is calibrated for a pre-mainnet aggregator: a genuine defect in the core best-execution ranking that persists to mainnet, but bounded per-swap dollar impact and no theft/loss vector.

### 6. Per-pool concentrated-liquidity quote failures are swallowed with no diagnostics

**CONFIRMED** · Severity: **Medium** · Kind: fact

- **Where:** packages/aggregator/src/discovery/concentratedLiquidityPools.mjs:257-271 (swallow at :269, null fallthrough at :289), completed by packages/aggregator/src/quotes/providers/concentratedLiquidity.mjs:104 treating null as "no quote" without onSourceError; telemetry at packages/api/src/handler.mjs:593-605
- **What:** In the live CL quoter provider every pool quote error is dropped: `quotePool({...}).catch(() => null)`, then nulls are filtered. Unlike the v2 path (runSourceQuote reports via onSourceError into /quote telemetry's sourceErrors), a decode failure, reverted quoter call, or RPC error inside quotePool produces an empty result with sourceErrorCount 0.
- **Why it matters:** An ACTIVE CL venue (muchfi-v3, barkswap) can stop quoting entirely — bad ABI, paused quoter, RPC regression — and the operator sees healthy telemetry while users silently get worse single-venue prices. This is the exact mechanism that would hide finding #2 in production.
- **Adversarial verifier:** Confirmed. The live CL quoter output provider wraps every per-pool quotePool call in .catch(() => null) (concentratedLiquidityPools.mjs:269) and quotePool contains all RPC reads (batched callMany of token0/token1/liquidity/slot0 + quoter call) and all decoding, so quoter reverts, bad-ABI decode failures, and promptly-rejecting RPC errors all become null; an all-fail venue returns null (line 289), which the adapter (quotes/providers/concentratedLiquidity.mjs:104) treats as "no quote" and returns [] without invoking onSourceError, bypassing the recordQuoteSourceError -> quoteDiagnostics -> telemetry.sourceErrors/sourceErrorCount pipeline (live.mjs:187, handler.mjs:603-604). The v2 contrast is accurate: v2Pools.mjs:132-152 wraps the whole per-source task in runSourceQuote so all v2 failures are reported. muchfi-v3 and barkswap-algebra are ACTIVE CL sources (registry.mjs:145, 294). No mitigations found: preQuoteVerifier is chain-id-only (live.mjs:241), verification/ has no quoter health checks, no test asserts per-pool failures surface, and KNOWN_ISSUES.md covers only the Solidity router. One narrowing nuance: a hung RPC (>1s) still surfaces via the adapter's 1s timeout race, so only fast failures are silent — but those (reverted quoter, decode error) are exactly the realistic venue-outage modes. Medium is calibrated: no fund-loss path, but for a best-price aggregator this silently removes an entire ACTIVE venue class while telemetry reports sourceErrorCount 0, degrading user prices invisibly.

### 7. Exact-output flow can strand approvals below the rebuilt maxAmountIn (approval and swap each re-quote independently)

**CONFIRMED** · Severity: **Low** · Kind: fact

- **Where:** packages/api/src/handler.mjs:628-645 (approval refresh + exact amount at line 638), handler.mjs:664-679 (independent /swap refresh), handler.mjs:292-335 (refreshSwapQuote has no cap vs. original maxAmountIn), packages/aggregator/src/swap/erc20Approval.mjs:58-83 (exact-amount approve), packages/aggregator/src/swap/venueCalldataBuilders.mjs:99-130 (encodes second refresh's maxAmountIn)
- **What:** /approval refreshes the quote and plans an exact approval for `amount = quote.maxAmountIn` (handler.mjs:638). /swap then refreshes AGAIN and builds calldata with a new maxAmountIn from the second refresh; if the price moved up between the two refreshes the approved allowance is below the encoded amountInMaximum, and there is no re-check.
- **Why it matters:** Exact-output swaps (API-supported, not yet in the web UI) intermittently fail simulation/execution with allowance errors after the user already paid for an approval tx; fail-closed but a guaranteed UX papercut and wasted gas once exactOutput ships in the UI.
- **Adversarial verifier:** Confirmed. /approval refreshes the quote and plans an exact (not unlimited) approval for the refreshed maxAmountIn (handler.mjs:638, erc20Approval.mjs:72-83); /swap then re-quotes independently via refreshSwapQuote (handler.mjs:292-335), which rebuilds maxAmountIn = new amountIn + inferred slippage with no comparison against the approved amount, and the exactOutput calldata builders encode that new maximum (venueCalldataBuilders.mjs:99-130). There is no allowance re-check: verifySwapTx only does eth_call+estimateGas and balancePreflight has no allowance logic. refreshSwapQuoteBeforeBuild defaults to true in production wiring (live.mjs:157). The only mitigation found — in-flight quote coalescing (handler.mjs:409-439, tested at handler.test.mjs:1004) — only dedupes concurrent requests and cannot span the approval-tx-confirmation gap in the real user flow; KNOWN_ISSUES.md covers contract-level venue allowances, not this. One nuance: venue routers pull only the actual required input, not the encoded amountInMaximum, so failure occurs only when the actual input at execution exceeds the originally approved maxAmountIn (price moved past the first refresh's slippage buffer) — i.e., intermittent rather than guaranteed on any upward move, consistent with the claim's "intermittently fail" wording. When triggered it fails closed via simulation (422 swap-not-buildable) after the user already paid for the approval tx. Web UI is exactInput-only (apps/web/src/lib/quote.js:130), so impact is limited to API consumers until exactOutput ships in the UI. Low severity is correctly calibrated: no fund-loss path, fail-closed, wasted approval gas + UX papercut only.
