# Code quality — backend

Part of the [2026-06-11 multi-agent repo audit](../README.md). Every finding below was independently challenged by an adversarial verifier agent that re-read the cited code; verdicts and calibrated severities are recorded inline.

**Result: 8 confirmed, 0 refuted.**

## Strengths

- Uniform dependency-injection style across the whole backend (client, fetchFn, nowMs, providers all injectable — e.g. createAggregatorApiHandler handler.mjs:379-408, createLiveAggregatorApiHandler live.mjs:134-158) yields a 250-test suite that runs in ~1.3s with zero network and full live-path coverage via fake RPC queues.
- Swap-build path has genuine defense in depth rather than trust-the-caller: createVerifiedCalldataBuilder re-verifies source status, ABI provenance, router-address match against the registry, and that emitted calldata starts with the verified selector (calldataRegistry.mjs:39-84), on top of buildSwapTx's chain/status/TTL/address re-checks (buildSwapTx.mjs:44-72).
- Failures on the quote path are degraded, not swallowed: per-source/per-provider errors return [] so healthy venues still quote, but every error is recorded into quoteDiagnostics and surfaced in /quote telemetry with per-source messages (live.mjs:112-132, handler.mjs:593-605, composite.mjs:63-79) — unusually good observability for a project this size.
- In-flight request deduplication for identical quote-candidate requests, including replaying captured source-errors into each concurrent caller's diagnostics and safe cleanup of the dedup map (handler.mjs:409-439), is careful concurrency engineering.
- The JSON-RPC client validates every result shape at the boundary (hex-quantity/hex-data/address asserts, safe-integer checks, batch responses re-matched by id — jsonRpcClient.mjs:1-57,126-136), and money math is consistently BigInt end-to-end with a bigint-aware JSON serializer (handler.mjs:20-22), closing most of the classic plain-JS numeric type-safety holes.

## Confirmed findings

### 1. One-hop route composition silently fails for V2 legs due to address-case mismatch in live wiring

**CONFIRMED** · Severity: **Medium** (auditor proposed High, calibrated by verifier) · Kind: fact

- **Where:** packages/aggregator/src/routes/oneHop.mjs:3-4 (strict === composition), wired with checksummed via token at packages/api/src/live.mjs:47-51,141,206-210 (from packages/config/src/tokens.mjs:5) against lowercased V2 legs from packages/aggregator/src/discovery/v2Pools.mjs:243-244 + packages/aggregator/src/quotes/adapters/v2.mjs:85-86
- **What:** composeOneHopCandidates uses strict equality on token addresses: `firstLeg.buyToken === viaToken && secondLeg.sellToken === viaToken` (oneHop.mjs:1-7). In the live wiring, viaToken comes from config in checksummed form — defaultOneHopViaTokens() returns OFFICIAL_DOGEOS_TOKENS WDOGE "0xF6BDB158A5ddF77F1B83bC9074F6a472c58D78aE" (live.mjs:47-51, tokens.mjs:5) — but the live V2 provider lowercases candidate tokens before building quotes (`normalizeAddress(sellToken...)` v2Pools.mjs:243-244, echoed into the candidate at v2.mjs:85-86). I reproduced this with the actual modules: with the checksummed via-token (exactly what live.mjs passes) the V2 first leg quotes successfully but ZERO one-hop candidates compose; with a lowercased via-token the identical setup composes 1 candidate. CL-provider legs echo input casing (concentratedLiquidity.mjs adapter passes through `sellToken`/`buyToken` raw), so only CL+CL hops ever compose. Tests mask this: oneHopRoutes.test.mjs uses symbolic tokens ("WDOGE" strings, lines 11/53/85), and live.test.mjs:457-526 stubs only the CL quoter, asserting `muchfi-v3+muchfi-v3` and never exercising a V2 leg.
- **Why it matters:** On the production /quote path, one-hop previews involving the muchfi-v2 venue (one of only three active venues) can never appear — USDC→USDT via WDOGE through the V2 pools is silently dropped with no error or diagnostic, so users see incomplete/worse route previews and /quote can report "no-route" when a v2 one-hop preview exists. Classic plain-JS hole a typed/normalized Address boundary would have caught.
- **Adversarial verifier:** Confirmed and reproduced with the actual modules: createLiveV2QuoteCandidateProvider lowercases sell/buy tokens before echoing them into candidates, while live.mjs passes the checksummed WDOGE config address as the one-hop via token, so canCompose's strict === never matches a V2 leg — checksummed via produced 0 one-hop candidates while the identical lowercased via produced 1 (muchfi-v2+muchfi-v2), with the V2 first leg quoting successfully and no error/diagnostic emitted. CL legs echo input casing raw (quotes/providers/concentratedLiquidity.mjs:128-129, adapters/concentratedLiquidity.mjs:83-84) so only CL+CL hops compose, matching the claim. Tests mask it exactly as cited (symbolic token strings in oneHopRoutes.test.mjs; live.test.mjs:457-526 stubs only the CL quoter). No mitigation exists: handler.mjs:131-132 lowercasing is only a cache key, and KNOWN_ISSUES.md/README document nothing about this. However, severity is downgraded from High to Medium: one-hop candidates are status "readOnly" previews that handler.mjs:622/658 hard-gate out of /swap (requires status "active"), so there is no fund-loss or mis-execution path — the user-visible impact is incomplete route previews and /quote returning "no-route" instead of "read-only" when the only path is a V2 hop via WDOGE (quoteService.mjs:205-206), on a testnet-only pre-deploy product. Real bug worth fixing by normalizing via tokens (or all candidate token fields) to lowercase at the boundary.

### 2. /quote maps all failures, including RPC/infrastructure errors, to HTTP 400 "invalid-quote-request"

**CONFIRMED** · Severity: **Medium** · Kind: fact

- **Where:** packages/api/src/handler.mjs:607-609 (catch-all for the /quote block at 547-610); RPC calls inside the try at handler.mjs:555 and 561 via live.mjs:241/245 and live.mjs:30-44; error string format at packages/dogeos-rpc/src/jsonRpcClient.mjs:84
- **What:** The entire /quote route body is wrapped in one try/catch that returns `errorResponse(400, "invalid-quote-request", error.message)` (handler.mjs:607-609). The try block includes `preQuoteVerifier` (live chain-id check via RPC, live.mjs:30-44) and `resolveProvider(gasPriceWei...)` which in live mode is `async () => client.getGasPriceWei()` (live.mjs:245). An RPC outage therefore surfaces as `400 invalid-quote-request: "eth_gasPrice failed with HTTP 503."`. By contrast /chain-status correctly distinguishes with 503 (handler.mjs:468-470). /approval similarly returns 422 for the server-side misconfiguration "No approval planner configured." (handler.mjs:614-616, 648-650).
- **Why it matters:** Clients and monitoring cannot distinguish user error from backend outage: an RPC failure looks like a malformed request (4xx), defeating alerting/retry logic and making real incidents look like client bugs. Raw internal error strings are also leaked to API consumers.
- **Adversarial verifier:** Confirmed verbatim: the /quote route wraps parsing, the live chain-id verification (preQuoteVerifier=verifyChain, live.mjs:30-44/241), and gas-price resolution (gasPriceWei=async () => client.getGasPriceWei(), live.mjs:245; invoked without guard via resolveProvider, handler.mjs:171-175) in one try/catch that returns 400 "invalid-quote-request" with the raw error.message (handler.mjs:607-608), and jsonRpcClient.mjs:84 produces exactly the claimed "eth_gasPrice failed with HTTP 503." string. Every other RPC-touching route correctly returns 503 (/chain-status 468-470, /venues 482, /intelligence 500, /verification 511, /activity 543), and /approval maps the server misconfig "No approval planner configured." to 422 (handler.mjs:614-616, 648-650), so /quote is a genuine outlier. The behavior is even cemented by a regression test (packages/api/test/live.test.mjs:277-304 asserts 400 invalid-quote-request for an RPC chain mismatch), and no acceptance is documented in README or KNOWN_ISSUES.md. Partial mitigation: RPC failures during candidate quoting are swallowed into telemetry diagnostics by the composite provider (composite.mjs:44-49, live.mjs:117-132) and return 200, so only the pre-quote chain check and gas-price fetch hit the 400 path — but a full RPC outage still surfaces as a client error on the product's core endpoint, defeating 5xx-based alerting/retries; raw error.message leakage is a repo-wide pattern (also on 503 routes), so the distinctive defect is the status-code misclassification. Medium is calibrated: no fund-loss path and trivial to fix, but it is a real operational defect that would persist into mainnet monitoring.

### 3. Verification snapshot provider has no in-flight deduplication; cold cache triggers a thundering herd of Blockscout+RPC fan-out

**CONFIRMED** · Severity: **Medium** · Kind: fact

- **Where:** packages/aggregator/src/verification/verificationSnapshot.mjs:865-877 (completion-only cache, no in-flight sharing, no negative cache); packages/api/src/handler.mjs:475,488,508 (three endpoints invoking provider per request); packages/aggregator/src/verification/verificationSnapshot.mjs:703-712 (per-target 3x Blockscout + RPC fan-out)
- **What:** createVerificationSnapshotProvider caches only on completion: `if (cached && now - cached.cachedAtMs <= cacheTtlMs) return cached.report;` then `await verifyDefaultSources(options)` with no in-flight promise sharing (verificationSnapshot.mjs:865-877). Three GET endpoints each call it per request (/venues handler.mjs:475, /intelligence handler.mjs:488, /verification handler.mjs:508). Each run does, per verification target (~20 in registry), 3 Blockscout HTTP fetches plus RPC read-checks and pool-state calls (verifySource, verificationSnapshot.mjs:703-712). Notably the quote path DID get in-flight dedup (resolveQuoteCandidates, handler.mjs:409-439) — the pattern exists but wasn't applied here. Failures are also never negative-cached, so a Blockscout outage makes every request re-run the full fan-out before returning 503.
- **Why it matters:** A single page load hitting /venues + /intelligence + /verification concurrently on an expired cache launches 3 full verification sweeps (~60+ outbound Blockscout requests at once), inviting rate-limiting from the only explorer and multi-second latency; under Blockscout downtime every incoming request repeats the whole sweep.
- **Adversarial verifier:** Confirmed against the code: createVerificationSnapshotProvider caches only after verifyDefaultSources resolves and shares no in-flight promise, so concurrent cold-cache requests each run a full sweep, and failures are never cached so a Blockscout outage makes every request repeat the sweep before returning 503. Production wiring (packages/api/src/live.mjs:227-237) adds no pre-warm or dedup wrapper. Scale verified by executing the registry: 24 verification targets x 3 Blockscout fetches = 72 outbound requests per sweep (bytecode is batched via getCodeMany, Blockscout calls are not), plus RPC read/pool-state checks. The asymmetry claim holds: the quote path has in-flight dedup (handler.mjs:409-439) and createChainVerifier in live.mjs dedups too. One correction to the consequence: the web app never calls /venues; the actual trigger is SettingsView.jsx:307 firing getIntelligence() and getVerification() concurrently (2 sweeps, ~144 Blockscout requests per Settings open on expired cache) — though with CORS * and no rate limiting all three endpoints are publicly hittable so the 3-sweep scenario is still reachable. No mitigations found: tests only exercise sequential TTL behavior, no KNOWN_ISSUES/README acceptance, default TTL 60s makes herds recurrent. Medium is calibrated for a pre-deploy service: no fund-loss, but a real unauthenticated ~72x outbound amplification against the chain's only explorer with a trivial fix already demonstrated elsewhere in the same file.
