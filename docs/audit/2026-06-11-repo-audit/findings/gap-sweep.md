# Gap-sweep findings (completeness critic round)

After the main dimensions completed, a completeness-critic agent identified missed areas and spawned extra finders. The run was stopped before these findings could be adversarially verified — treat them as **unverified** leads, not confirmed findings.

## Area: Unbounded in-memory cache grow…

- Strength: The quote-path instance of the same provider is bounded by design: it uses the default `payloadProvider = estimatedSwapPayloadForFee`, which returns one of only three fixed strings derived from `ESTIMATED_SWAP_PAYLOAD_BYTES` (v2/v3/algebra) or '0x' (l1GasPriceOracle.mjs:5-9,45-49), so its cache keyspace is at most 3 entries — no leak there.
- Strength: The cache value is small and the read-side TTL logic is correct (entries older than 15s are treated as misses and re-fetched), so the data served is never stale; the only defect is the missing eviction/cap on the write side.
- Strength: Input validation is consistently defensive across the fee/calldata path (assertHexData on payloads and calldata, positiveUint/positiveBigInt guards in venueCalldataBuilders.mjs and buildSwapTx.mjs), which limits the shape of what can be cached to well-formed hex.

### 1. Unbounded in-memory Map in swap-path data-finality fee provider grows forever (slow leak + unauthenticated memory-DoS)

**UNVERIFIED** · Severity: **High** · Kind: fact

- **Where:** packages/aggregator/src/fees/l1GasPriceOracle.mjs:65,73,76-78,89; packages/api/src/live.mjs:166-171,268-275; packages/aggregator/src/swap/verifySwapTx.mjs:51,64-72; packages/aggregator/src/swap/venueCalldataBuilders.mjs:89-95,119-131; packages/api/src/handler.mjs:653-708
- **What:** `createDogeosDataFinalityFeeProvider` allocates `const cache = new Map()` (l1GasPriceOracle.mjs:65). The 15s `cacheTtlMs` is checked only on READ (`if (cached && now - cached.cachedAtMs <= cacheTtlMs) return cached.feeWei;`, lines 76-78); on miss it inserts `cache.set(cacheKey, { feeWei, cachedAtMs: now })` (line 89). There is no `cache.delete`, no `cache.clear`, no size cap anywhere in the file (grep for delete/clear/maxSize returns nothing) — a stale entry is never removed, only overwritten if the exact same key recurs. The swap instance is created with `payloadProvider: ({ transaction }) => transaction.data` (live.mjs:166-171) and wired into `verifySwapTransaction` (live.mjs:268-275), which calls `resolveDataFinalityFeeWei(dataFinalityFeeWei, { transaction, ... })` on the freshly built swap calldata (verifySwapTx.mjs:64-72). That calldata is effectively unique per request: builders encode amountIn/minAmountOut/recipient AND a wall-clock `deadline` (e.g. buildV2SwapExactTokensForTokensCalldata, venueCalldataBuilders.mjs:89-95; deadline flows in via buildSwapTx.mjs routeBinding and handler.mjs:287). The cache key is the lowercased full calldata hex (`const cacheKey = payload.toLowerCase()`, line 73), so each POST /swap (handler.mjs:653-708) with a different amount/deadline permanently adds a new entry holding the full ~460-char calldata string. The process is a single long-running node server (web/src/server.mjs, Dockerfile CMD `node packages/web/src/server.mjs`).
- **Why it matters:** Under normal traffic the Map grows monotonically (each distinct swap adds ~0.5-1 KB: a ~460-char latin1 key string plus a {BigInt,number} value and Map slot overhead) and is never reclaimed — a guaranteed slow leak that ends in OOM on the single production process. Worse, POST /swap is unauthenticated, so an attacker scripting varied amountIn/deadline values adds a new permanent entry on every request; at ~1000 req/s that is ~0.5-1 MB/s of unreclaimable heap, driving the only production process to OOM in minutes-to-hours (a remote, unauthenticated crash of the whole web+API server). Minimal fix: cap the cache (LRU or hard `maxEntries` with eviction on insert), and/or evict-on-read when an entry is past TTL, and/or key by a fixed-width hash of the payload (e.g. sha256) instead of the raw calldata so the value object — not the full calldata — dominates and a bounded keyspace policy is enforceable.

### 2. Oracle outage silently zeroes the data-finality fee used in scoring and swap verification

**UNVERIFIED** · Severity: **Medium** · Kind: fact

- **Where:** packages/aggregator/src/fees/l1GasPriceOracle.mjs:58,91-94; packages/api/src/live.mjs:161-171
- **What:** On any RPC failure the provider swallows the error and returns zero: `catch (error) { onProviderError?.(error, input); return BigInt(fallbackFeeWei); }` (l1GasPriceOracle.mjs:91-94) with the default `fallbackFeeWei = 0n` (line 58). Neither live-handler instance passes `onProviderError` (live.mjs:161-165 quote path, 166-171 swap path), so the error is dropped with no logging or signal. The resulting fee feeds route scoring (resolvedDataFinalityFeeWei is passed to the v2/concentrated-liquidity quote candidate providers, live.mjs:178,185) and the swap verification readout (verifySwapTx.mjs:82-84 sets `verification.dataFinalityFeeWei`).
- **Why it matters:** During an L1 gas-price oracle outage the data/finality fee silently collapses to 0, so route scoring under-counts true cost (can flip which venue wins) and the verification response reports a fee of 0 with no error surfaced — a correctness/observability gap that can mislead users and routing right before a real-money mainnet deploy. Fix: pass an `onProviderError` that logs/metrics the failure, and consider failing closed (propagate the error or use a conservative non-zero fallback) rather than returning 0n.

## Area: Zero HTTP security headers on …

- Strength: runtime-config.js injection is XSS-hardened: runtimeConfigScript JSON-stringifies and escapes `<` via `.replaceAll("<", "\\u003c")` and freezes the object (packages/web/src/server.mjs:45-52), so a hostile DOGEOS_CLIENT_ID env value cannot break out into script context.
- Strength: serveStatic has a real path-traversal guard: decode → normalize → strip leading ../ → resolve, then verifies `absolutePath.startsWith(root + sep)` before reading (packages/web/src/server.mjs:99-110) — many hand-rolled static servers get this wrong.
- Strength: The TradingView library is self-hosted from apps/web/src/public rather than loaded from a third-party CDN (ChartView.jsx:28-29), which keeps the future CSP script-src to 'self' and removes a live remote-script dependency.
- Strength: Cache-control: no-store on all HTML/JS/runtime-config responses (server.mjs:124, 140) means a future header/CSP fix or config rotation propagates immediately with no stale-cache window.
- Strength: The swap path is robust to in-page staleness by design: the backend re-quotes inside /approval and /swap before building calldata, and the client validates chain id 6281971 + connected account first (apps/web/src/lib/execute.js:13-17), limiting what a UI-layer attacker gains from serving stale route data.

### 3. Production web server sends zero HTTP security headers on every response (HTML shell, JS, runtime-config, API)

**UNVERIFIED** · Severity: **Medium** · Kind: fact

- **Where:** packages/web/src/server.mjs:120-127, packages/web/src/server.mjs:136-144, packages/api/src/handler.mjs:11-16, apps/web/src/index.html:1-20, vite.config.mjs:82-85, README.md:32-37
- **What:** serveStatic returns only `"cache-control": "no-store"` and `"content-type": CONTENT_TYPES[extname(filePath)] ?? "application/octet-stream"` (packages/web/src/server.mjs:120-127); serveRuntimeConfig the same pair (server.mjs:136-144); the 500 handler only content-type (server.mjs:163-165). The API sets only JSON_HEADERS = CORS triple + content-type on all routes including OPTIONS preflight (packages/api/src/handler.mjs:11-16, 444-445). apps/web/src/index.html has no <meta http-equiv="Content-Security-Policy">. The Vite dev plugin adds nothing (vite.config.mjs:82-85), and the README's recommended nginx config is bare `proxy_pass http://127.0.0.1:8080; proxy_set_header Host $host;` with no add_header lines (README.md:32-37); .env.example has no header guidance either. Repo-wide grep for CSP/X-Frame-Options/nosniff/Referrer-Policy/Permissions-Policy/HSTS/helmet finds zero hits in source.
- **Why it matters:** For a wallet dApp the missing headers are an entire defense-in-depth layer, and nothing downstream (proxy guidance, meta tags) compensates. Concrete recommended set for server.mjs (add to a shared header object merged into every response) — frame-ancestors 'none' (+ X-Frame-Options: DENY for legacy), X-Content-Type-Options: nosniff (the octet-stream fallback at server.mjs:125 is sniffable without it), Referrer-Policy: strict-origin-when-cross-origin (full swap-state URLs currently leak to fonts.googleapis.com/fonts.gstatic.com), Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=(), and HSTS (Strict-Transport-Security: max-age=31536000) at the nginx/Caddy TLS layer with a README note. Starting CSP compatible with the app's verified needs (self-hosted TradingView script+iframe per ChartView.jsx:28-29, Google Fonts per index.html:8-10, same-origin /runtime-config.js classic script per index.html:17, React inline style props, wagmi RPC + WalletConnect per sdkConfig.js:8): default-src 'self'; script-src 'self'; worker-src 'self' blob:; frame-src 'self'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data:; connect-src 'self' https://rpc.testnet.dogeos.com https://dogeos-testnet-public.unifra.io https://blockscout.testnet.dogeos.com wss://ws.rpc.testnet.dogeos.com wss://relay.walletconnect.com https://*.walletconnect.com https://*.walletconnect.org https://sdk.dogeos.com; frame-ancestors 'none'; base-uri 'self'; form-action 'self'; object-src 'none'. Ship it as Content-Security-Policy-Report-Only first and exercise the chart + Connect Kit (the SDK/WalletConnect endpoint list is the part most likely to need additions) before enforcing.

### 4. App is frameable by any origin — clickjacking against the in-page approval/swap confirm flow

**UNVERIFIED** · Severity: **Medium** · Kind: fact

- **Where:** apps/web/src/ui/SwapFlow.jsx:649-661, apps/web/src/lib/execute.js:8-16, packages/web/src/server.mjs:120-127
- **What:** No frame-ancestors directive or X-Frame-Options exists anywhere (see finding 1 citations), so https://evil.example can iframe the deployed app. The swap confirmation is an ordinary in-page button: `<button onClick={confirm} ... >confirm swap` (apps/web/src/ui/SwapFlow.jsx:649-661); clicking it immediately drives the approval-then-swap transaction pipeline (apps/web/src/lib/execute.js:8-16 documents the confirm path: /approval send + /swap send).
- **Why it matters:** An attacker page can overlay a transparent iframe so a user's click lands on "confirm swap" (or the approve step) with attacker-staged parameters. The wallet's own confirmation popup is a real partial mitigation — the final signature happens outside the page — but users routinely approve a wallet prompt they believe they initiated, and the iframe can hide/cover the page context that would reveal the mismatch. Since the app never legitimately needs to be embedded, `frame-ancestors 'none'` + `X-Frame-Options: DENY` closes this class entirely at zero compatibility cost. Severity Medium today (testnet, low traffic), but this specific header should be considered a mainnet blocker for a wallet dApp.

### 5. No CSP containment for the unpinned ~26MB vendored TradingView bundle that runs same-origin next to the wallet bridge

**UNVERIFIED** · Severity: **Medium** · Kind: judgment

- **Where:** apps/web/src/ui/ChartView.jsx:28-29, scripts/fetch-charting-library.sh:18, apps/web/src/lib/execute.js:22-24, apps/web/src/lib/execute.js:349-368
- **What:** ChartView loads the vendored library as a first-party script: `const SCRIPT_SRC = \`${LIBRARY_PATH}charting_library.standalone.js\`` (apps/web/src/ui/ChartView.jsx:28-29). That bundle is restored at build time by `git clone --depth 1 "${REPO_URL}"` from the mutable, commit-unpinned mirror `https://github.com/worldofhacks/advanced_charting_library.git` (scripts/fetch-charting-library.sh:18,37) — README requires running it on every fresh deploy (README.md:21-22) and the Dockerfile path does too (README.md:62). With no CSP (finding 1), that script — or any future XSS — has unrestricted same-origin run of the page: the wallet bridge `window.dogeosAggregatorWallet` (apps/web/src/lib/execute.js:22-24) through which all eth_sendTransaction requests flow, plus localStorage swap history `doge.history` (execute.js:349-368) and settings/chart keys (apps/web/src/ui/useSettings.js:38-59, ChartView.jsx:256).
- **Why it matters:** The supply-chain exposure (unpinned mirror) is a separate known finding; the point here is that headers provide none of the usual blast-radius containment. A tampered bundle could silently swap recipient/spender addresses in the UI before the user ever sees the wallet prompt, or initiate plausible-looking approval requests via the bridge — and exfiltrate via fetch to any origin since there is no connect-src. The starting CSP in finding 1 (script-src 'self', connect-src allowlist, object-src 'none') would not stop a fully malicious same-origin bundle from executing, but the connect-src allowlist blocks the easy exfiltration paths and a CSP with reporting gives detection; pinning the mirror to a commit hash is the complementary fix. Consequence chain is judgment; every cited mechanism is verified fact.
