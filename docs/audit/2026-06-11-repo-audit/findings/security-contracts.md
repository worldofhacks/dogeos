# Security — Solidity contracts

Part of the [2026-06-11 multi-agent repo audit](../README.md). Every finding below was independently challenged by an adversarial verifier agent that re-read the cited code; verdicts and calibrated severities are recorded inline.

**Result: 6 confirmed, 0 refuted.**

## Strengths

- Per-execute in-memory balance-delta Ledger (DogeSwapRouter.sol:186-208, 237-242) is a genuinely strong design: every spend/payout/refund is measured as current-balance minus entry-snapshot, native is seeded excluding msg.value (line 176), and explicit amounts must be <= delta — making pre-existing/airdropped/stranded funds structurally unspendable via execute. I adversarially traced double-spend-across-commands, mid-stream snapshot, donation-inflation, and Permit2-sentinel-token paths and found no accounting hole.
- Permit2 integration follows the UniversalRouter pattern correctly: owner/from is hardwired to msg.sender in both _permit2Permit and _permit2TransferFrom (DogeSwapRouter.sol:226, 232) and p.spender == address(this) is enforced (line 224), with an explicit third-party-allowance-drain regression test (RouterPermit2.t.sol::test_thirdParty_cannotDrainVictimAllowance).
- Movement-only 7-command whitelist with immutable venue addresses and no arbitrary-call/delegatecall command (Commands.sol:8-14, _dispatch DogeSwapRouter.sol:210-219); settlement (fee -> minOut -> payout -> refunds) is enforced after the command loop regardless of the command program, making minOut a contract guarantee (DogeSwapRouter.sol:287-302).
- Chain-facts-driven engineering for a zkEVM target: EIP-1153/PUSH0/MCOPY confirmed by direct on-chain opcode probe BEFORE adopting ReentrancyGuardTransient, ecrecover/precompile support matrix checked against DogeOS docs, and venue bytecode presence verified (audit/CHAIN_FACTS.md:35-88) — exactly the diligence most teams skip on non-mainnet EVMs.
- Deploy script self-verifies Permit2 CREATE2 determinism: uses the canonical vanity salt (explicitly documenting that salt 0 yields a wrong address, DeployRouter.s.sol:33-39) and asserts computed address == canonical before proceeding (lines 67-74); guarded-launch caps are set and feeBps==0 asserted in the same broadcast (lines 97-104), so the router is never live-and-uncapped.
- Verification depth is real, not claimed: I ran the suite — 52/52 local tests pass (the 53rd is the fork-gated test), including 7 stateful Foundry invariants at 25,600 calls each with 0 reverts, plus Echidna/Medusa property files; receive() restricted to WDOGE, guardian strictly pause-only, Ownable2Step everywhere.
- Honest, well-triaged audit pack: SLITHER_TRIAGE.md dispositions each finding with in-source one-line justifications for every suppression, and KNOWN_ISSUES.md accurately characterizes accepted trade-offs (standing venue allowance, revert-on-receive recipients, cap governance) rather than hiding them.

## Confirmed findings

### 1. recipient == address(0) silently disables settlement in production execute — strands all pulled funds with a successful tx

**CONFIRMED** · Severity: **Medium** (auditor proposed High, calibrated by verifier) · Kind: fact

- **Where:** packages/contracts/src/DogeSwapRouter.sol:288 (with lines 40-44, 159, 177, 181-182, 243-246); packages/aggregator/src/swap/balancePreflight.mjs:3-15; packages/aggregator/src/swap/buildSwapTx.mjs:1-5; packages/contracts/audit/KNOWN_ISSUES.md:10-21 (§1 premise), §4 (gap)
- **What:** In _settle: "if (s.recipient == address(0)) return; // no-op (unit tests only)" (DogeSwapRouter.sol:288). The NatSpec admits this is "used only by unit tests" (lines 41, 44), but nothing on-chain enforces it. With recipient=0 the early return skips the fee, the minOut check, the payout, AND the entire refund loop — so every token pulled via PERMIT2_TRANSFER_FROM plus any msg.value stays in the router, the tx SUCCEEDS, and Swapped is emitted (line 182). The stranded funds become 'pre-existing balance' (unspendable via execute, by design) and are recoverable only via owner rescue() behind the TimelockController (>=48h per DEPLOYMENT.md). The off-chain stack does not guard this either: assertHexAddress accepts 0x0000…0000 — "/^0x[0-9a-fA-F]{40}$/" (packages/aggregator/src/swap/balancePreflight.mjs:11-13). It also quietly undermines KNOWN_ISSUES.md §1's load-bearing premise that "the router holds ~zero balance between transactions", which is what bounds the standing type(uint256).max venue allowances granted by _approveVenue (DogeSwapRouter.sol:243-246).
- **Why it matters:** A single zero-address default anywhere in the JS quote/encode pipeline (a classic ABI-encoding bug class) makes user swaps 'succeed' while users receive nothing and their funds sit in the router until a timelocked governance rescue — a silent fund-strand incident, not a revert. Not in KNOWN_ISSUES.md. Trivial fix pre-mainnet: revert on recipient == address(0) and move the no-op path into a test harness subclass.
- **Adversarial verifier:** Every cited fact verified: _settle's early return on recipient==address(0) (line 288, exact comment "no-op (unit tests only)") skips fee, minOut, payout, and the refund loop while execute still succeeds and emits Swapped (line 182); nothing on-chain enforces the "tests only" intent; stranded funds become ledger-unspendable pre-existing balance recoverable only via owner rescue() behind the intended TimelockController (24-48h, KNOWN_ISSUES §7); both JS address validators accept 0x0000...0000; the gap is not documented in KNOWN_ISSUES.md (§4 covers only guardian/feeRecipient/constructor zero-checks; INVARIANTS.md I1 mentions the mode only to exclude it from fuzzing); and a recipient=0 strand does undermine §1's "~zero balance between transactions" premise backing the standing type(uint256).max venue allowances (lines 243-246). However, severity is downgraded from High to Medium: it is not attacker-exploitable (recipient is msg.sender-supplied in their own tx, so only the caller's funds strand), it requires a separate off-chain zero-default bug to manifest, funds are recoverable (delayed, not lost), and crucially the current JS pipeline never encodes DogeSwapRouter execute()/Settlement at all (zero references in packages/aggregator, packages/api, apps/web — calldata targets venue routers directly), so no live code path can hit it today on a pre-deploy testnet project. It remains a real, must-fix-pre-mainnet trap: it silently converts a classic ABI zero-default bug class from a revert into a "successful" fund-strand with a misleading Swapped event, and four test suites depending on the no-op mode confirm the proposed fix (revert on zero recipient; move the no-op into a test harness) is the right shape.

### 2. Registry repoint is instant (Safe-direct) while router governance is timelocked — trust asymmetry in the canonical discovery pointer

**CONFIRMED** · Severity: **Medium** · Kind: judgment

- **Where:** packages/contracts/src/DogeSwapRegistry.sol:34-38 (purpose at :9-15); packages/contracts/audit/DEPLOYMENT.md:206-215; packages/contracts/script/DeployRouter.s.sol:107-112
- **What:** DogeSwapRegistry.setCurrentRouter is plain onlyOwner with owner = the Safe directly: "function setCurrentRouter(address router) external onlyOwner { currentRouter = router; ... }" (DogeSwapRegistry.sol:34-38). DEPLOYMENT.md §6b confirms "The Safe accepts directly (no timelock)" (DEPLOYMENT.md:206-209). The registry's stated purpose is for the web app/integrators to discover the live router (DogeSwapRegistry.sol:9-15) — i.e., the address users will sign Permit2 PermitSingle approvals for as spender. Router config changes get a 24-48h public delay, but the pointer deciding where user approvals are directed does not. I verified nothing in the repo consumes currentRouter yet (grep over apps/web, packages/aggregator, packages/api, packages/config), so this is forward-looking.
- **Why it matters:** Once integrators/the web app read the registry (its stated purpose), a compromised solo-founder Safe key can instantly repoint users to a malicious 'router' that drains every new Permit2 approval — exactly the attack class the timelock exists to delay and publicize. Either put the registry behind the same timelock or document it as a hot pointer that front-ends must pin/verify. Not covered in THREAT_MODEL.md (scoped to 'single-chain router only').
- **Adversarial verifier:** Every cited fact verified against the code. DogeSwapRegistry.setCurrentRouter is plain onlyOwner (DogeSwapRegistry.sol:34-38) and the deploy script hands the registry to the Safe directly (DeployRouter.s.sol:109 registry.transferOwnership(routerSafe)) while the router goes to the timelock (line 112); DEPLOYMENT.md:208 states verbatim "The Safe accepts directly (no timelock)". The registry's own NatSpec (lines 9-15) declares its purpose as the discovery pointer for "the web app, indexers, partner front-ends" — i.e., the address future UIs will put in Permit2 PermitSingle.spender (the legit router enforces spender == address(this) at DogeSwapRouter.sol:224, so approvals are bound to whatever address the UI sources). The auditor's forward-looking qualifier is accurate: grep over apps/web, packages/aggregator/api/config/web and scripts finds zero consumers of currentRouter/DogeSwapRegistry today. No mitigation exists anywhere: KNOWN_ISSUES.md and THREAT_MODEL.md contain zero registry mentions (THREAT_MODEL.md:10 scopes to "single-chain router only"), DEPLOYMENT.md gives no pin/verify guidance to integrators, and the registry NatSpec's "does NOT move funds" framing (line 15) understates rather than accepts the risk — repointing doesn't move funds, but it redirects all subsequent approvals. The asymmetry is real: a compromised solo-founder Safe gets a 24-48h public delay for router config changes but an instant, silent repoint of the pointer deciding where new user approvals are spendable. Medium is correctly calibrated — not High (pre-deploy, testnet-only, no consumers yet, requires Safe compromise) but not Low, since the compromised-key scenario is exactly what the project's own timelock design treats as in-scope, the registry's sole documented purpose is to be consumed, and the fix is cheap (timelock the registry or document it as a hot pointer that front-ends must pin/verify, plus add it to KNOWN_ISSUES.md).

### 3. Venue ABI compatibility only weakly verified: single V3 fork test that skips on revert; no fork coverage for Barkswap Algebra or MuchFi V2

**CONFIRMED** · Severity: **Low** (auditor proposed Medium, calibrated by verifier) · Kind: judgment

- **Where:** packages/contracts/test/fork/RouterFork.t.sol:99-121 (skip-on-revert reference leg; only fork test, V3 only); packages/contracts/src/interfaces/IAlgebraSwapRouter.sol:6-15; packages/contracts/src/interfaces/IUniswapV3SwapRouter.sol:6-14; also packages/contracts/audit/DEPLOYMENT.md:265 (runbook accepts "PASS / clean SKIP" as deploy gate)
- **What:** RouterFork.t.sol tests only the MuchFi V3 leg, and its reference call is wrapped in "try ... exactInputSingle(...) ... catch { console2.log(\"SKIP: direct V3 WDOGE->USDC swap reverted\"); vm.skip(true); }" (RouterFork.t.sol:99-115) — an ABI/struct mismatch with the live venue would surface as a clean SKIP, not a failure. There is no fork test at all exercising the Barkswap Algebra router, whose interface uses the nonstandard Algebra-Integral struct with a user-supplied `deployer` field (IAlgebraSwapRouter.sol:6-15), nor MuchFi V2. The V3 interface is SwapRouter02-shaped (no deadline field, IUniswapV3SwapRouter.sol:6-14); if any deployed venue uses the other variant, the selector differs and every swap on that venue reverts.
- **Why it matters:** Failure mode is atomic revert (liveness, not fund loss) because the router is immutable — a venue interface mismatch discovered post-deploy means that venue is dead for the contract's lifetime and requires a redeploy + registry migration. Cheap to close: un-skippable fork assertions (or recorded mainnet-style differential txs) for all three venues before mainnet.
- **Adversarial verifier:** All cited facts verified accurate: RouterFork.t.sol contains a single test exercising only the MuchFi V3 leg, its reference call is wrapped in try/catch with vm.skip(true) so a live ABI mismatch surfaces as a clean SKIP, and there is no fork coverage for Barkswap Algebra or MuchFi V2; DEPLOYMENT.md:265 even blesses "PASS / clean SKIP" as the deploy gate, and KNOWN_ISSUES.md does not document the gap. Selector math confirms the mechanics (cast sig: V3 struct = 0x04e45aaf, Algebra struct = 0x1679c792, deadline variant = 0x414bf389), so a venue on the wrong variant would atomically revert every swap on that venue forever (immutable router → redeploy + registry migration). However, severity should be downgraded from Medium to Low: substantial mitigation exists outside the contracts package — docs/dogeos-router-verification-runbook.md (2026-06-01 live snapshot) and the verification pipeline (verificationSnapshot.mjs, scripts/verify-dogeos-sources.mjs) confirmed selectors 0x04e45aaf and 0x1679c792 are present in the live deployed venue bytecode, relationship reads (factory()/WETH9()/poolDeployer()) match, and live eth_call/eth_estimateGas swap simulations passed for all three venues — byte-identical struct shapes to what DogeSwapRouter encodes, making the feared mismatch unlikely for the pinned venues. Impact is liveness-only (no fund-loss path; router holds ~zero balance; post-deploy /swap simulation would exclude a dead venue rather than burn user gas), the project itself rates the analogous "venue address wrong → fail to swap, redeploy" state as LOW (KNOWN_ISSUES.md §4), and the fix (un-skippable fork assertions for all three venues pre-mainnet) is cheap. Real as a test-process judgment finding, but Low severity for this pre-deploy, testnet-only project.

### 5. Swapped event carries no token/amount/fee data

**CONFIRMED** · Severity: **Low** · Kind: judgment

- **Where:** packages/contracts/src/DogeSwapRouter.sol:88 (declaration), packages/contracts/src/DogeSwapRouter.sol:182 (emit)
- **What:** "event Swapped(address indexed sender, address indexed recipient);" (DogeSwapRouter.sol:88), emitted once per execute (line 182). No buyToken, gross/net output, fee taken, or input tokens are logged; fee payments and refunds are visible only via token Transfer events and traces.
- **Why it matters:** On mainnet, incident forensics (e.g., investigating a stranded-funds report), fee-revenue reconciliation, and the project's own /activity indexing must fall back to trace-level analysis on a chain whose explorer tooling (Blockscout testnet) may not expose traces well. Adding (buyToken, amountOut, fee) to the event is free at this pre-deploy stage but impossible after, since the router is immutable.
- **Adversarial verifier:** Verified verbatim: event Swapped(address indexed sender, address indexed recipient) at line 88, emitted once per execute at line 182 with no buyToken/amount/fee data. The claim is actually slightly understated: for native-DOGE outputs, _settle/_pay (lines 287-308) deliver fee and payout via raw call{value:}(""), producing no log at all — so native fee revenue is reconstructible only from traces/internal txs. No mitigation exists anywhere: KNOWN_ISSUES.md has no acceptance of the minimal event; CODE_MATURITY.md omits the Trail of Bits "Auditing" (event emission/monitoring) category from its 9-category scorecard entirely; THREAT_MODEL.md/DEPLOYMENT.md say nothing about monitoring or forensics; the design plan mirrors the same minimal event without rationale. One minor overstatement: the /activity endpoint (packages/api/src/handler.mjs:80-99) uses Blockscout's address-transactions REST listing, not Swapped-event decoding, so it does not break today — but it also cannot show decoded amounts/fees, and richer events are the only trace-free path to that. Contract is confirmed immutable (no proxy; "upgrades are a fresh deployment"), so the event schema is a one-way door at mainnet deploy. Low severity is correctly calibrated: no fund-loss path, but a real, permanent observability/fee-reconciliation gap that is free to fix pre-deploy.

### 6. TimelockController deployed with the Safe holding DEFAULT_ADMIN_ROLE

**CONFIRMED** · Severity: **Low** · Kind: judgment

- **Where:** packages/contracts/script/DeployRouter.s.sol:85
- **What:** "TimelockController timelock = new TimelockController(timelockMinDelay, proposers, executors, routerSafe);" (DeployRouter.s.sol:85) — the 4th arg grants routerSafe the timelock's DEFAULT_ADMIN_ROLE, letting the Safe grant/revoke proposer/executor/canceller roles with no delay (operations themselves still respect minDelay; updateDelay remains self-administered). OZ docs recommend renouncing this role so role management also flows through the delay.
- **Why it matters:** Combined with the solo-founder single-Safe setup, a compromised Safe can silently rewire timelock roles immediately rather than over a publicly visible delayed operation, weakening the monitoring value of the timelock. Low because scheduled operations still cannot be executed early; consider admin = address(timelock) or address(0) at mainnet deploy.
- **Adversarial verifier:** Confirmed. DeployRouter.s.sol:85 passes routerSafe as the TimelockController's 4th constructor arg, and the vendored OZ v5.6.1 (lib/openzeppelin-contracts/contracts/governance/TimelockController.sol:115-137) grants that address DEFAULT_ADMIN_ROLE, which (with no _setRoleAdmin overrides) is the role-admin for PROPOSER/EXECUTOR/CANCELLER — so the Safe can grant/revoke those roles with no delay. The claim's caveats are also accurate: updateDelay remains self-only (lib line 447-451) and scheduled operations still enforce minDelay. No mitigation exists: grep for renounce/DEFAULT_ADMIN/TIMELOCK_ADMIN outside lib/ returns nothing; audit/DEPLOYMENT.md:11,152 documents "proposer/executor/admin = ROUTER_SAFE" as intended config without flagging the implication or a renounce step; KNOWN_ISSUES.md has no entry; test/RouterGovernance.t.sol:29 replicates admin=safe and only tests delay on router ops. Minor quibble: "silently" overstates it since RoleGranted/RoleRevoked events are emitted, but role changes do bypass the schedule→execute monitoring window. Since the same Safe already holds proposer+executor+canceller, the admin role adds no fund-loss or early-execution path — Low (judgment/hardening, matching OZ's recommendation to renounce or pass address(0)/address(timelock) at mainnet deploy) is correctly calibrated.
