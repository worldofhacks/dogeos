# Dependencies, DevEx & operations

Part of the [2026-06-11 multi-agent repo audit](../README.md). Every finding below was independently challenged by an adversarial verifier agent that re-read the cited code; verdicts and calibrated severities are recorded inline.

**Result: 8 confirmed, 0 refuted.**

## Strengths

- Zero-runtime-dependency backend: every import across packages/{aggregator,api,web,config,dogeos-rpc}/src resolves to node: builtins only — the quoting/swap-calldata path that real funds will depend on cannot be poisoned by any npm package at runtime. This is an unusually strong supply-chain posture; the only gap is that the Dockerfile doesn't exploit it (finding above).
- wagmi ^2.19.5 is NOT a dead dependency, contrary to suspicion: @dogeos/dogeos-sdk declares peerDependencies "wagmi": ">=2.0.0" (node_modules/@dogeos/dogeos-sdk/package.json), so it must be installed even though no first-party file imports it. The project even pins this contract in a test: packages/web/test/staticApp.test.mjs:103-104 asserts the exact SDK version "3.2.0" and the wagmi range — dependency drift is test-enforced, which is rare and good.
- The four npm overrides (axios 1.16.1, bn.js 5.2.3, protobufjs 7.6.2, ws 8.21.0) are deliberate, correct supply-chain triage: npm ls confirms all four are pulled solely via @dogeos/dogeos-sdk → @tomo-inc/cubist-sig-sdk and resolve everywhere to the pinned safe versions ('deduped'/'overridden'), with package-lock.json committed.
- Test suite quality: 250 tests pass in ~1.6s using only node:test — no jest/vitest/framework dependency at all, so the missing-CI fix is pure wiring with nothing to install or configure.
- .env.example is complete and accurate against actual code reads (HOST, PORT, DOGEOS_CLIENT_ID, WALLETCONNECT_PROJECT_ID + VITE_ variants — verified via repo-wide process.env/import.meta.env grep), and it correctly documents the runtime-vs-build-time split. The /runtime-config.js injection (packages/web/src/server.mjs:37-52) lets the wallet client id change with a restart instead of a rebuild — good ops design for a registered-domain credential.
- The hand-rolled static server shows security care: path-traversal guarded by a resolved-root prefix check (server.mjs:99-111) and the runtime config script is XSS-escaped via replaceAll("<", "\\u003c") (server.mjs:49).
- Docker layer caching is done correctly (package.json + lockfile copied and npm ci before COPY . .) and .dockerignore is thorough (.env files, contracts build artifacts, fuzzing corpora, dist all excluded).
- Contracts CI is genuinely strong for a solo project: forge test with invariants at 512 runs / 200 depth plus a Slither job with fail-on: config (.github/workflows/contracts-security.yml) — the JS side just needs the same treatment.

### 1. No CI for the JS test suite, and no linter/formatter/typecheck anywhere

**CONFIRMED** · Severity: **Medium** (auditor proposed High, calibrated by verifier) · Kind: fact

- **Where:** /Users/quietguy/Documents/Dev/dogeos/.github/workflows/contracts-security.yml:2-4; /Users/quietguy/Documents/Dev/dogeos/package.json:5-14
- **What:** The only workflow is contracts-security.yml, path-filtered to contracts: "on: pull_request: { paths: [\"packages/contracts/**\", ...] }". package.json scripts contain only build/test/dev/start entries — no lint, no typecheck — and no eslint/prettier/biome config or tsconfig exists anywhere in the repo (verified by find). The 250-test suite passes locally in 1.6s (`npm test`: "# pass 250 ... duration_ms 1579") but is never executed on push or PR.
- **Why it matters:** Every change to the quoting/route-selection/swap-tx-building JS (the code that decides what calldata users sign) can merge and deploy with failing tests and nobody would know. As a solo founder there is no human reviewer either, so CI is the only safety net — and wiring it is ~10 lines (checkout + node 22 + npm ci + npm test) for a suite that already runs in 1.6s.
- **Adversarial verifier:** Confirmed in full: contracts-security.yml is the sole workflow and is path-filtered to packages/contracts/** (plus the workflow file itself), so the JS suite never runs on push/PR; package.json scripts contain no lint/format/typecheck; the only eslint/prettier/tsconfig hits are vendored third-party code (openzeppelin lib, permit2 deps, .tmp/ethers, bundled charting library); no git hooks, no husky/lefthook/pre-commit, no alternative CI (CircleCI/GitLab/Travis/Jenkins/Vercel/Netlify), and the Dockerfile does not run tests. Reproduced the suite: 250 pass, 0 fail, ~1.3s, so the claimed ~10-line fix is accurate. No documented acceptance anywhere (README.md:67 only says to run npm test manually; audit docs cover contracts only). Severity downgraded from High to Medium: this is a missing process safety net rather than an exploitable flaw on a pre-deploy testnet-only repo, and DogeSwapRouter.sol provides a partial on-chain backstop (enforced settlement minAmountOut + aggregate cap) that converts most silent JS regressions into wrong quotes or reverted txs rather than direct fund loss — though minAmountOut itself is computed by the untested-in-CI JS, so the backstop is incomplete. Given solo founder with no human reviewer, this should be treated as a hard blocker before mainnet deploy, at which point it would warrant High.

### 2. Docker build clones a non-public, unpinned GitHub repo to inject the charting library

**CONFIRMED** · Severity: **Medium** (auditor proposed High, calibrated by verifier) · Kind: fact

- **Where:** /Users/quietguy/Documents/Dev/dogeos/Dockerfile:19-20; /Users/quietguy/Documents/Dev/dogeos/scripts/fetch-charting-library.sh:18,36; /Users/quietguy/Documents/Dev/dogeos/.dockerignore:4
- **What:** Dockerfile runs "RUN bash scripts/fetch-charting-library.sh && npm run build:web" (Dockerfile:19-20). The script does `git clone --depth 1 "${REPO_URL}"` with REPO_URL defaulting to https://github.com/worldofhacks/advanced_charting_library.git (fetch-charting-library.sh:18,36) — no commit pin, no checksum. I verified the repo returns HTTP 404 (private or gone), and the Dockerfile has no ARG/secret mechanism for git credentials. .dockerignore explicitly excludes the local vendored copy (apps/web/src/public/advanced_charting_library), so the build MUST clone.
- **Why it matters:** Three concrete consequences: (1) `docker build` fails on any machine/CI/cloud builder without ambient GitHub credentials — the image as committed is unbuildable by anyone but the founder's logged-in laptop; (2) when it does build, it ships whatever HEAD of an external repo contains, un-reviewed and un-pinned, into JS served to swap users — a single supply-chain injection point (e.g. recipient-address-swapping code) once real funds flow on mainnet; (3) TradingView's Advanced Charts license prohibits redistribution, so mirroring the licensed bundle on GitHub is a license exposure the script's own comment ("~26MB, licensed... intentionally NOT committed") shows awareness of, then recreates one repo over.
- **Adversarial verifier:** Confirmed in full. Dockerfile:19-20 runs the fetch script with no ARG/secret/ssh-mount mechanism; the script (lines 18, 36) does an unpinned, checksum-free `git clone --depth 1` of https://github.com/worldofhacks/advanced_charting_library.git; .dockerignore:4 (plus .gitignore:5) excludes the vendored copy so the script's early-exit can never fire in a Docker build — the clone is mandatory. I verified the repo is private: anonymous git smart-HTTP info/refs returns 401 (web/API return 404); a sanitized git ls-remote only succeeded via this host's ambient credentials, proving the "founder's laptop only" dynamic — and in fact even the founder's `docker build` would fail, since build containers inherit no host git credentials, making the committed Dockerfile unbuildable by anyone as-is (the README's Docker instructions are broken as written). No mitigations exist: no tests cover the script, no risk acceptance in packages/contracts/audit/KNOWN_ISSUES.md or README, and scripts/verify-repository-scope.mjs merely allowlists the gitignored paths. Severity calibrated down from High to Medium for the current pre-deploy testnet posture: consequence (1) is an ops defect with an easy founder workaround (non-Docker build path on a credentialed machine, and there is no JS CI to break); consequence (2) — un-reviewed, un-pinned external HEAD shipping into DEX frontend JS — is a genuine supply-chain injection point but requires compromising the founder's own private GitHub repo/account and only becomes a fund-loss path at mainnet (should be fixed pre-mainnet by pinning a commit SHA and verifying a sha256, at which point it would rate High); consequence (3), license exposure from the private mirror, is Low.

- **Why it matters:** A DEX frontend's install step (`npm ci` in the Docker build) executes lifecycle scripts from ~2,170 packages, almost all pulled for wallet chains (Sui/Aptos/TON/Expo) the app never touches — a single compromised transitive poisons the built frontend. The exact-pinned SDK plus exact-pinned overrides also means every SDK upgrade re-litigates whether the overridden versions still satisfy the new tree; the override list is already demonstrably incomplete (8 high vulns remain). Worth raising with the DogeOS SDK team and/or import-splitting before mainnet.
- **Adversarial verifier:** Verified end-to-end: package.json:16 pins @dogeos/dogeos-sdk 3.2.0 and lines 26-31 override exactly axios/bn.js/protobufjs/ws; the lockfile has 2,172 node_modules entries (2,173 packages incl. root); node_modules measures exactly 2.0GB; npm ls traces all four overridden packages through @dogeos/dogeos-sdk → @tomo-inc/embedded-wallet-providers → @tomo-inc/oidc-auth → @tomo-inc/cubist-sig-sdk (one hop the auditor elided), which bundles Aptos/Cosmos/Solana/TON/OKX/Keplr libs plus the full Expo CLI via isomorphic-webcrypto; and npm audit --omit=dev reports verbatim "49 vulnerabilities (16 low, 25 moderate, 8 high)" including the cited valibot ReDoS via @mysten/sui and the @expo/config chain. Two overstatements worth noting: only 14 lockfile entries actually have hasInstallScript (so "lifecycle scripts from ~2,170 packages" exaggerates install-time execution, though all 2,172 remain supply-chain surface), and npm ci against the committed lockfile freezes the tree, partially mitigating "a single compromised transitive" (requires the pinned version itself to be malicious or a lockfile regen). Most of the 49 vulns sit in Sui/Aptos/TON/Expo code paths the app never executes and Vite tree-shaking likely keeps them out of the bundle, so there is no current exploit — the risk is latent. No mitigation or documented acceptance exists anywhere (no .npmrc/ignore-scripts; KNOWN_ISSUES.md is contracts-only; README only says testnet-only). ### 4. Production servers log nothing: errors are swallowed, no request logs, no health endpoint

**CONFIRMED** · Severity: **Medium** · Kind: fact

- **Where:** packages/web/src/server.mjs:162-172 (and identical pattern at packages/api/src/server.mjs:58-70); startup-only console lines at packages/web/src/server.mjs:199 and packages/api/src/server.mjs:111; Dockerfile:22-29 has no HEALTHCHECK
- **What:** The web server's top-level catch returns the error to the client but never logs it: `} catch (error) { response.writeHead(500, ...); response.end(JSON.stringify({ error: { code: "web-server-error", message: error.message } }))` (server.mjs:162-172). Grep across packages/api and packages/web finds exactly two console statements total — the startup lines (web server.mjs:199, api server.mjs:111). No /health route exists anywhere (grep), and the Dockerfile has no HEALTHCHECK.
- **Why it matters:** When the RPC flaps or quoting starts throwing in production, the operator has zero signal — no error line, no request trace, nothing to grep — and container orchestrators cannot restart an unhealthy instance. Internal error.message strings are also leaked to clients while being hidden from the server's own logs, the exact inverse of what an operator wants. For a solo-run service about to handle real funds, a console.error in that catch block plus a /health route is the minimum.
- **Adversarial verifier:** Confirmed verbatim: the web server's top-level catch returns error.message to the client with no logging, the API server has the identical silent catch (server.mjs:58-70), and grep verifies exactly two console statements across packages/api+packages/web (both startup lines), no logging library, and no process-level uncaughtException/unhandledRejection handlers. No /health route exists (handler.mjs:710 404-falls-through) and the Dockerfile has no HEALTHCHECK. The pattern is actually worse than claimed: handler.mjs also contains two fully silent promise swallows (.catch(() => {}) at lines 435 and 567) and ~10 catch blocks that forward error.message to clients without any server-side record. Only partial mitigation found: a /chain-status route returns 503 on RPC failure and could serve as an ad-hoc orchestrator probe, so "orchestrators cannot restart an unhealthy instance" is mildly overstated — but nothing in README, .env.example, or audit docs (KNOWN_ISSUES.md is contracts-only) documents this or accepts the logging gap. Medium is well calibrated: no direct fund-loss path (non-custodial quote/tx-building server), but total operational blindness plus client-side leakage of internal error strings is a real gap for a solo-run service heading to mainnet.

### 6. cache-control: no-store on every static asset, including content-hashed bundles and the ~26MB charting library

**CONFIRMED** · Severity: **Medium** · Kind: fact

- **Where:** /Users/quietguy/Documents/Dev/dogeos/packages/web/src/server.mjs:120,124
- **What:** serveStatic sets `"cache-control": "no-store"` unconditionally for all static responses (server.mjs:124) and buffers each file fully in memory per request via `await readFile(filePath)` (server.mjs:120). This covers Vite's content-hashed immutable assets and the TradingView library the fetch script itself describes as "~26MB" (fetch-charting-library.sh:6).
- **Why it matters:** Every page view re-downloads multi-megabyte chart bundles from a single Node process — slow loads for users, multiplied bandwidth cost, and a cheap accidental self-DoS at launch traffic. Hashed Vite assets are safe to serve with `immutable, max-age=31536000`; only index.html and runtime-config.js need no-store.
- **Adversarial verifier:** Confirmed verbatim: serveStatic sets "cache-control": "no-store" unconditionally (server.mjs:124) and buffers every file fully via await readFile (server.mjs:120, plus a second full copy in writeFetchResponse at line 95). dist/advanced_charting_library measures exactly 26MB (matching fetch-charting-library.sh:6) and is served by this server. One nuance: chart bundles load lazily only when the user opens the chart panel (SwapView.jsx:903 chartDocked gate, ChartView.jsx:51-60), so "every page view re-downloads chart bundles" is slightly overstated — but the finding actually understates the routine cost, since every page view re-downloads the content-hashed 13MB sdk-wallet-provider-B1-5d05h.js Vite bundle (dist/assets = 14MB total) with no ETag/Last-Modified to allow revalidation. No mitigation anywhere: README's nginx example is a bare proxy_pass with no caching, no CDN is documented, no test asserts the no-store header (it is an unconsidered default, not a pinned decision), and KNOWN_ISSUES.md is contracts-only. Medium is calibrated: real launch performance/bandwidth/availability issue for a single Node process double-buffering multi-MB files per request, but not a security or fund-loss path, and the proposed fix (immutable max-age=31536000 for hashed /assets/, no-store only for index.html and runtime-config.js) is correct and trivial.
