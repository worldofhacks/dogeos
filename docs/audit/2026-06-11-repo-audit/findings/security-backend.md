# Security — backend/API

Part of the [2026-06-11 multi-agent repo audit](../README.md). Every finding below was independently challenged by an adversarial verifier agent that re-read the cited code; verdicts and calibrated severities are recorded inline.

**Result: 4 confirmed, 0 refuted.**

## Strengths

- Rigorous input validation before any value reaches an RPC call or calldata: jsonRpcClient.normalizeTransactionRequest enforces hex address/data/quantity on every eth_call/eth_getCode/estimateGas param (jsonRpcClient.mjs:43-57, 178-200), and buildSwapTx/erc20Approval/v2Pools independently re-validate addresses and positive amounts. Arbitrary token addresses also get filtered out by pinned-pool matching (filterSourcesByTokenPair) before reaching RPC, so untrusted addresses don't drive unbounded RPC calls.
- Static file serving has solid path-traversal containment: decode -> normalize -> strip leading ../ -> resolve against root -> reject unless absolutePath === root or starts with root+sep (packages/web/src/server.mjs:99-111), with API paths matched against an explicit allowlist Set rather than prefix logic.
- /runtime-config.js exposes only the intentionally-public DOGEOS_CLIENT_ID and WALLETCONNECT_PROJECT_ID, and escapes '<' to '<' to prevent </script> breakout (packages/web/src/server.mjs:45-52). No other env vars are reachable; no secret leakage path found.
- /activity is not an open proxy/SSRF: the address is strictly regex-validated (isHexAddress) before URL construction, the Blockscout base URL is fixed server-side config (not client-controllable), and limit is clamped to 1..50 (handler.mjs:47-55, 515-545).
- Execution safety on /swap: refreshSwapQuoteBeforeBuild re-quotes live on-chain so a client-supplied quote's amounts can't poison the built transaction, calldata comes only from the verified calldata registry builders, and the server never signs — it returns an unsigned tx the user signs in their own wallet (handler.mjs:653-707, live.mjs:256-276).
- Per-source (1000ms) and per-provider (1500ms) timeouts via Promise.race bound the RPC latency of a single quote request and prevent one slow venue from hanging the whole response (v2Pools.mjs:132-152, composite.mjs:27-42).

## Confirmed findings

### 1. Public HTTP servers have no request-body size limit, no rate limiting, and /quote amplifies into multi-source RPC fan-out (DoS)

**CONFIRMED** · Severity: **Medium** (auditor proposed High, calibrated by verifier) · Kind: fact

- **Where:** packages/api/src/server.mjs:25-32 (readIncomingBody) & :95-103 (createServer/listen); packages/web/src/server.mjs:72-79 (readIncomingBody); packages/api/src/handler.mjs:127-138 (quoteCandidateRequestKey includes amountIn) & :547 (POST /quote) & :409-439 (in-flight dedup keyed on that key); packages/aggregator/src/routes/oneHop.mjs:89-103 (nested Promise.all fan-out)
- **What:** Both production servers read the entire request body into memory with no cap: `message.on("data", (chunk) => chunks.push(chunk)); message.on("end", () => resolve(Buffer.concat(chunks)))` and then `await request.json()`. `createServer(...).listen(...)` sets no `requestTimeout`/`headersTimeout`/`maxRequestsPerSocket` and there is no rate limiter. A single `POST /quote` fans out: block-number fetch + the `direct` provider (v2 + concentrated-liquidity adapters) AND the `one-hop` provider, which itself does `Promise.all(usableViaTokens.map(...))` first-leg then nested `Promise.all` second-leg quotes — each issuing `eth_call`/batch RPC reads. The in-flight de-dup cache is keyed on `amountIn` (`quoteCandidateRequestKey` includes `amountIn?.toString()`), so an attacker bypasses it by varying the amount by 1 wei per request, forcing fresh RPC fan-out every time. README only 'recommends' a reverse proxy for TLS, not for rate limiting.
- **Why it matters:** An unauthenticated client can (a) exhaust server memory with one large POST body, and (b) cheaply amplify each request into many upstream RPC calls, exhausting/rate-limiting or running up cost on the DogeOS RPC endpoint and taking quoting offline for all users. For a server about to front real-money swaps, the quote path is the availability-critical surface.
- **Adversarial verifier:** Every cited fact checks out. Both servers read the full request body into memory unbounded: server.mjs lines 25-32 (api) and 72-79 (web) do exactly the quoted `message.on("data", chunk => chunks.push(chunk)); on("end", () => resolve(Buffer.concat(chunks)))`, and the body is read in `requestFromIncomingMessage` BEFORE the handler runs, so no downstream validation can cap it. `createServer(...).listen(...)` (server.mjs:95-103, web 183-191) sets no requestTimeout/headersTimeout/maxRequestsPerSocket, and a repo-wide grep finds no rate limiter anywhere in JS source. POST /quote (handler.mjs:547) does fan out: resolveQuoteCandidates -> quoteCandidateProvider (direct v2 + concentrated-liquidity + one-hop), and oneHop.mjs:89-103 is exactly the Promise.all(usableViaTokens.map(...)) first-leg followed by nested Promise.all second-leg, each leg an RPC read. The in-flight de-dup (handler.mjs:409-439) is keyed by quoteCandidateRequestKey which includes amountIn?.toString() (line 133), so varying amountIn by 1 wei produces a distinct key and bypasses coalescing, forcing fresh fan-out per request — claim is correct. No mitigation exists: parseQuoteRequest validates chain/mode/positivity but not body size or request rate; KNOWN_ISSUES.md and THREAT_MODEL.md cover only contract-level DoS (native-recipient revert), not the HTTP surface; README only recommends a reverse proxy for TLS/clientId, not rate limiting. So the finding is real and the consequence (memory exhaustion via one large POST; cheap RPC amplification exhausting the shared DogeOS RPC endpoint and taking quoting offline) is plausible. I downgrade severity from High to Medium: this is a testnet-only, pre-deploy, non-custodial service with no fund-loss path (DoS affects availability only), and the standard remediation — body-size cap, server timeouts, and a rate-limiting reverse proxy — is well understood and squarely a pre-mainnet hardening item rather than a fundamental design flaw. It should be fixed before fronting real-money swaps on mainnet.

### 2. Production Docker image runs the public-facing server as root

**CONFIRMED** · Severity: **Low** (auditor proposed Medium, calibrated by verifier) · Kind: fact

- **Where:** /Users/quietguy/Documents/Dev/dogeos/Dockerfile:2,22-29 (no USER directive anywhere in the 29-line file)
- **What:** The Dockerfile has no `USER` directive, so `CMD ["node", "packages/web/src/server.mjs"]` runs as root (the default user of `node:22-slim`). The image even installs `git`/`bash` and binds `HOST=0.0.0.0 PORT=8080` to all interfaces. The base image ships a non-root `node` user that is unused.
- **Why it matters:** The internet-exposed Node process and any RCE/file-write bug in it (or its deps) executes with root inside the container, maximizing blast radius of a compromise. Adding `USER node` is a one-line defense-in-depth hardening expected before a mainnet-adjacent deploy.
- **Adversarial verifier:** Confirmed as fact: the Dockerfile uses node:22-slim (line 2, defaults to root; ships an unused non-root `node` user), has no USER directive, installs git/bash into the runtime image (single-stage build, lines 6-8), binds HOST=0.0.0.0 PORT=8080 (lines 22-23), and runs the public server as root via CMD (line 29). No mitigation exists anywhere: KNOWN_ISSUES.md/DEPLOYMENT.md cover contracts only with zero container mentions, README documents docker run as a supported deploy path with no caveat, and there is no compose/user override. The fix is trivially safe — the server does no runtime filesystem writes and 8080 is unprivileged, so `USER node` is a drop-in one-liner. Severity downgraded Medium→Low: the consequence is purely conditional defense-in-depth (amplifies a hypothetical future RCE in a stateless, keyless static-file/API-proxy server; container escape needs a separate runtime/kernel bug), which audit convention rates Low/Informational — though it should be fixed before any mainnet-adjacent deploy since Docker is the documented production path.

### 3. Error responses and the 500 fallback echo raw internal error.message to clients

**CONFIRMED** · Severity: **Low** · Kind: fact

- **Where:** packages/api/src/handler.mjs:469 (also 482, 500, 511, 543, 608, 649, 706); packages/api/src/server.mjs:58-70; packages/web/src/server.mjs:162-172
- **What:** Every route returns the upstream error string verbatim, e.g. `errorResponse(503, "chain-status-unavailable", error.message)` (and the same for venues/intelligence/verification/activity), and both servers' catch-all returns `JSON.stringify({ error: { ... message: error.message } })` with status 500. Upstream errors here originate from `fetch`/undici and the JSON-RPC client, whose messages can include the internal RPC host/connection details (e.g. ECONNREFUSED with address) on infrastructure failure.
- **Why it matters:** Leaks internal infrastructure topology (RPC endpoint/host, error internals) to any caller, aiding reconnaissance. Impact is limited here because the RPC URL is public testnet config, but the pattern should be replaced with generic public messages + server-side logging before mainnet.
- **Adversarial verifier:** Confirmed verbatim at every cited location: errorResponse(503, ..., error.message) for chain-status/venues/intelligence/verification/activity (handler.mjs:469/482/500/511/543, plus 608/649/706 for quote/approval/swap), and both servers' catch-alls return JSON.stringify({error:{...message: error.message}}) at 500 (api/src/server.mjs:58-70, web/src/server.mjs:162-172). errorResponse (handler.mjs:31-41) sends the message verbatim to clients. The leak path is real: jsonRpcClient.mjs:89/133 echoes upstream RPC-node error bodies, and verificationSnapshot.mjs:485 throws `Request failed ${status}: ${url}` embedding the upstream request URL into 503 responses. One nuance: Node 22 undici fetch failures have message "fetch failed" with ECONNREFUSED/address details in error.cause (not serialized), so the specific ECONNREFUSED-with-address example is overstated, but URL/upstream-error leakage holds via other paths. No mitigation exists — no sanitization layer, no documented acceptance in KNOWN_ISSUES.md or README, and handler tests (handler.test.mjs:775/801) assert the verbatim echo as intended behavior. Low severity is correctly calibrated: leaked endpoints are public testnet config (e.g. https://blockscout.testnet.dogeos.com), no secrets exposed, recon-only impact; replacing with generic messages plus server-side logging before mainnet is the right recommendation.

### 4. CORS allow-origin * is applied to all endpoints including POST /quote, /approval, /swap

**CONFIRMED** · Severity: **Low** · Kind: judgment

- **Where:** packages/api/src/handler.mjs:11-16 (JSON_HEADERS with access-control-allow-origin: *), :444-445 (OPTIONS preflight returns JSON_HEADERS); applied to POST /quote (:547), /approval (:612), /swap (:653) via jsonResponse (:24-29)
- **What:** `JSON_HEADERS` hard-codes `"access-control-allow-origin": "*"` and is returned on every response, including the state-shaping POST endpoints and the OPTIONS preflight (`return new Response(null, { status: 204, headers: JSON_HEADERS })`).
- **Why it matters:** Any third-party website can script-call the quote/approval/swap builders. This is acceptable for this design (no cookies/session/credentials, all outputs are unsigned txns the user must sign in their own wallet) so there is no CSRF/credential-theft vector — but combined with the missing rate limiting (above) it widens the set of origins that can drive the RPC-amplification DoS. Worth an explicit allowlist once a known frontend origin exists.
- **Adversarial verifier:** Confirmed as described (Phase 2.1: full attacker control of origin/body — wildcard CORS, unconditional 204 preflight, no auth/allowlist/rate limit in packages/api/src or packages/web/src/server.mjs; Phase 2.2/2.3: N/A — not a bounds or concurrency issue, though the inFlightQuoteCandidates dedup at handler.mjs:421-438 slightly narrows duplicate-quote load). Every route including POST /quote, /approval, /swap returns JSON_HEADERS via jsonResponse(), so any third-party webpage can script-call the builders and read responses. Adversarial check: CORS is browser-enforced only — curl/bot DoS ignores it — so the consequence is capped at conscripting visitors' browsers for distributed load and silent third-party API consumption; the finding itself correctly disclaims CSRF/credential theft (no cookies/sessions, no allow-credentials, outputs are unsigned txns requiring the user's own wallet signature). No documented acceptance found in README, docs/, or packages/contracts/audit/. Judgment-kind finding with accurate facts and plausible (marginal) consequence: TRUE POSITIVE, Low correctly calibrated for a testnet-only pre-deploy project; explicit origin allowlist before mainnet is the right fix.
