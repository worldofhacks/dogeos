# Code quality — frontend

Part of the [2026-06-11 multi-agent repo audit](../README.md). Every finding below was independently challenged by an adversarial verifier agent that re-read the cited code; verdicts and calibrated severities are recorded inline.

**Result: 8 confirmed, 0 refuted.**

## Strengths

- useQuote.js is an unusually well-engineered async state machine for hand-rolled code: monotonic seq-guard drops stale responses (useQuote.js:87-111), AbortController per request, debounce + 10s poll with the stale-poll-timer correctly cleared in finally (useQuote.js:122-129), seq bump on idle so in-flight responses can't resurrect a cleared quote (useQuote.js:144-151), and a separate 1s expiry ticker computed from expiresAtMs so the countdown stays honest even if a poll is late. useTokenBalances applies the same seq-guard discipline (useTokenBalances.js:66-80).
- Rigorous 'honest data' discipline throughout the UI: no fabricated USD values, price impact, sparklines, or candles — each omission is documented in-place with HONESTY comments and replaced by real data or an explicit placeholder (SwapView.jsx:14-16/397/859-876, TokensView.jsx:8-11, ChartView.jsx:5-12 with a real-ticks-only datafeed and a truthful 'building live price history' overlay plus graceful fallback when the 26MB chart library isn't vendored).
- Theme consistency is genuinely good despite heavy inline styles: every color in every view flows from theme.js tokens via useTheme() (no stray hex outside theme.js except intentional brand constants), light/dark + 4 accents with onAccent contrast handling for light accents (theme.js:46), and prefers-reduced-motion support in global.css:98-103.
- lib/execute.js is a clean framework-agnostic execution lifecycle: chain-id + account-mismatch guards before sending (execute.js:137-184), receipt polling with revert detection and timeout (execute.js:216-241), and a thoughtful raw-error -> user-message mapping covering faucet funding, slippage reverts, deadline, paused venues, and wallet rejection (execute.js:93-132).
- Failure-mode UX in the wallet connect path is deliberately handled rather than swallowed: useWallet.js documents the prior silent-rejection bug it fixes (useWallet.js:26-29), every connect failure surfaces an actionable toast (runInjectedConnect, useWallet.js:144-158), and the multi-wallet EIP-6963 chooser handles the 0/1/N injected-provider cases explicitly (useWallet.js:186-208).
- Comment quality is exceptional for a solo project: nearly every file opens with a contract-style header explaining provenance ('ported from app.js'), behavior guarantees ('seq-guard', 'pending is NON-dismissible'), and the WHY of decisions — making the 8k-line frontend far more auditable than its size suggests.

## Confirmed findings

### 1. Components defined inside SwapView's render body remount the amount input on every keystroke

**CONFIRMED** · Severity: **High** · Kind: fact

- **Where:** apps/web/src/ui/SwapView.jsx:226 (Chip), apps/web/src/ui/SwapView.jsx:256 (Section), apps/web/src/ui/SwapView.jsx:364-396 (input rendered inside <Section no="01">); also apps/web/src/ui/SwapFlow.jsx:329 (Big) and apps/web/src/ui/ActivityView.jsx:285 (Row inside entries.map); aggravated by apps/web/src/ui/useQuote.js:170-178 (1s setInterval re-render)
- **What:** `const Chip = ({ token, side }) => ...` (SwapView.jsx:226) and `const Section = ({ no, title, right, children }) => (...)` (SwapView.jsx:256) are declared inside the SwapView function body, so they get a new function identity on every render. The 'you pay' `<input value={payAmt} onChange={(e) => setPayAmt(...)}>` (SwapView.jsx:379-396) is rendered as a child of `<Section no="01">` (SwapView.jsx:364). Every keystroke calls setPayAmt -> re-render -> new `Section` element type -> React unmounts and recreates the entire section subtree including the input DOM node. The same anti-pattern repeats with `Big` inside SwapFlow (SwapFlow.jsx:329) and `Row` inside ActivityView's map (ActivityView.jsx:285).
- **Why it matters:** React reconciles by element type reference; a new arrow function per render means the subtree is torn down and remounted on each parent render. For the main swap amount field this loses focus/caret after each typed character (and dismisses the mobile keyboard), making multi-character entry on the core trading input broken or severely degraded. It also defeats all transition styles inside those subtrees and wastes DOM work on every quote poll re-render.
- **Adversarial verifier:** Confirmed at every cited line: Chip (SwapView.jsx:226) and Section (SwapView.jsx:256) are plain arrow functions declared inside SwapView's render body with no memoization, and the controlled pay-amount input (value={payAmt}, onChange -> setPayAmt, lines 379-396) is a child of <Section no="01"> at line 364, with payAmt held as SwapView state (line 128). React reconciles by element-type reference, so each keystroke produces a new Section identity and React unmounts/remounts the whole section subtree including the input DOM node — losing focus/caret per character (canonical "component defined inside component" remount bug); there is no autoFocus or ref-based focus restore. The finding actually understates it: useQuote.js:170-178 ticks setSecondsLeft every 1s while a quote is ready, so the input also remounts and drops focus every second between keystrokes. Secondary citations (Big in SwapFlow.jsx:329, Row in ActivityView.jsx:285 inside entries.map) are accurate too, though those subtrees contain no focusable input so impact is perf/transition-only. No mitigations found: apps/web has zero tests, and neither README nor packages/contracts/audit/KNOWN_ISSUES.md documents/accepts this. Severity High is calibrated for a frontend finding — no funds at risk, but the core trading input of the product is functionally broken for multi-character entry.

### 2. useWallet has no initial-state seeding; tab switches remount views into a 'disconnected' state where connect silently no-ops

**CONFIRMED** · Severity: **High** · Kind: fact

- **Where:** apps/web/src/ui/useWallet.js:117-141 (no seeding) and :173 (silent early return); remount trigger at apps/web/src/ui/Shell.jsx:200 and :293
- **What:** useWallet starts at EMPTY and only updates from `dogeos:sdk-wallet-updated` window events (useWallet.js:124-141). Neither bridge replays state to late subscribers: the injected bridge publishes only on initialize/openModal/disconnect/accountsChanged (injected-wallet.js:358-372, 517-521), and the SDK bridge publishes only when its own effect deps change (sdk-wallet-provider.jsx:123-173). The hook never seeds from the bridge's synchronous getters (`getAddress`/`isConnected` exist at injected-wallet.js:509-516 but are unused on mount). Shell remounts each view with `key={view}` (Shell.jsx:200, Shell.jsx:293), and SwapView/TokensView/ActivityView each call their own useWallet() (SwapView.jsx:68, TokensView.jsx:25, ActivityView.jsx:71). Worse, connect() early-returns with no UI feedback when the bridge says it is already connected: `if (state.address || wallet.isConnected?.()) return;` (useWallet.js:173).
- **Why it matters:** After connecting and switching to any other tab and back, SwapView/ActivityView see address='' until the next bridge event: the swap CTA shows 'connect wallet', balances read 0, on-chain activity never loads — while the Shell header chip (whose older hook instance got the event) still shows connected. Clicking 'connect wallet' then silently does nothing because of the early return, leaving the user stuck with no recovery path short of a page reload or a wallet-side event. Also, disconnect() only resets the local hook instance's state (useWallet.js:231-236); sibling instances rely on the injected bridge's own publish to catch up.
- **Adversarial verifier:** Confirmed on every cited line. useWallet seeds from useState(EMPTY) and updates only via the 'dogeos:sdk-wallet-updated' window event (useWallet.js:118, 124-141), never reading the bridge's synchronous getAddress/isConnected getters (injected-wallet.js:509-516, sdk-wallet-provider.jsx:134-138). Neither bridge replays state: publishWalletState is an uncached CustomEvent dispatch (sdk-wallet.jsx:25-27), the injected bridge publishes only on initialize/openModal/disconnect/provider events (injected-wallet.js:358-372 plus call sites), the SDK bridge only on effect-dep changes (sdk-wallet-provider.jsx:123-173), and preloadSdkWallet early-returns once loaded (useWallet.js:91) so a remount triggers nothing. Shell remounts views via key={view} (Shell.jsx:200 desktop, 293 mobile) and SwapView.jsx:68/TokensView.jsx:25/ActivityView.jsx:71 each own a useWallet instance, so after connect + tab switch the new view sees address='' — SwapView CTA reverts to 'connect wallet' (SwapView.jsx:176, 590), balances query with owner='' (:147-148), ActivityView never loads (:104-108) — while the Shell chip stays 'connected'. Clicking connect then silently no-ops at useWallet.js:173 because the bridge's isConnected() is true (no toast, no publish). No mitigations: no wallet context, no replay/focus refresh, no hook-level tests (packages/web/test/injectedWallet.test.mjs covers only the bridge), nothing in KNOWN_ISSUES.md (contracts-only) or README. Minor correction to the claim: a recovery path besides reload/wallet-side events exists — clicking the header chip disconnects via the bridge, which publishes and resyncs all instances, after which reconnect works — but it is non-obvious. High is calibrated: no fund-loss path, but the core swap/balances/activity flow breaks deterministically on routine tab navigation with a silently dead connect button, which is shipping-blocking for a DEX frontend heading to mainnet.

Token/source catalog failures are likewise empty-caught with no user-visible state (SwapView.jsx:90-92, 102).

- **Adversarial verifier:** Confirmed on the code as written: useQuote returns status/error (useQuote.js:186-194, set at 119-121) but SwapView.jsx:166 destructures only quote/isScanning/isReady/secondsLeft/refresh, and a grep shows zero error/fail/retry strings anywhere in SwapView — there is no error state on the primary screen. Medium is calibrated: no fund-loss or security path, but for a DEX heading to mainnet, routine backend failures rendering the primary screen indistinguishable from a hung app is a real pre-launch UX defect.
