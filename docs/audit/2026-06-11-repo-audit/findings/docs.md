- **Where:** README.md:3, README.md:10 (contradicted by README.md:75); packages/contracts/README.md:3 + :44; "audit" resolves to self-prepared audit-prep package per packages/contracts/README.md:34 and packages/contracts/audit/THREAT_MODEL.md:3

The "audit" being referenced is the self-prepared audit-prep package (packages/contracts/audit/), not a third-party audit.
- **Why it matters:** In DeFi, "audited" is read as third-party security audit. Anyone skimming the first paragraph (a user deciding to trust funds, an integrator, or a journalist) gets a materially false impression that the bottom-of-page disclaimer only partially corrects. For a project heading to a real-money mainnet deploy, this wording will be quoted against the founder if funds are ever lost. Rewording to "audit-prepped" or "with a full internal security program" preserves the claim honestly.
- **Adversarial verifier:** Confirmed verbatim. The only audit artifact in the repo is the self-prepared package: packages/contracts/README.md:34 calls it "Audit-prep package: `audit/`" and audit/THREAT_MODEL.md:3 states "Methodology: Trail of Bits audit-prep" — no third-party report exists. Partial mitigations: the in-file disclaimer (bottom of both READMEs) and the contracts README's more honest "Security program"/"Audit-prep" framing — but these do not neutralize the unqualified "audited" in the root README's first sentence, which in DeFi convention implies a third-party audit. No KNOWN_ISSUES.md entry acknowledges the wording. Severity Medium is calibrated: no fund-loss path today (testnet-only, pre-deploy), but the contracts self-describe as "Mainnet-grade" and the project is heading to a real-money deploy where this headline claim becomes materially misleading; fix is a one-word reword (e.g., "audit-prepped" or "internally audited").

### 4. Server silently falls back to serving raw apps/web/src when dist/ is missing, contradicting README

**CONFIRMED** · Severity: **Low** · Kind: fact

- **Where:** /Users/quietguy/Documents/Dev/dogeos/packages/web/src/server.mjs:33-35 (roots at :9-10, default wired at :148); /Users/quietguy/Documents/Dev/dogeos/README.md:29
- **What:** README.md:29 says the server "serves the built app from `apps/web/dist/`". But packages/web/src/server.mjs:33-35 defines: "function defaultStaticRoot() { return existsSync(resolve(DEFAULT_DIST_ROOT, \"index.html\")) ? DEFAULT_DIST_ROOT : DEFAULT_SOURCE_ROOT; }" — if dist/index.html is absent it serves the unbuilt source tree (raw JSX, untranspiled imports) instead of failing.
- **Why it matters:** An operator who runs `node packages/web/src/server.mjs` directly (e.g. a systemd unit copied from the Dockerfile CMD) without a prior `vite build` gets a 200-serving but broken app with no error, and the README gives no hint this fallback exists. Either document the fallback or make missing dist/ a hard startup error in production.
- **Adversarial verifier:** Confirmed. defaultStaticRoot() (server.mjs:33-35) silently falls back to apps/web/src/ when dist/index.html is absent, with no env override, no log, and no startup error; README.md:29 states only that the server "serves the built app from apps/web/dist/" and the fallback is documented nowhere (README, docs/, KNOWN_ISSUES.md all silent). The consequence is real: apps/web/src/index.html exists and loads /main.jsx as a module, but CONTENT_TYPES (server.mjs:25-31) has no .jsx entry so it ships as application/octet-stream — browsers refuse the module, yielding a 200 index with a blank app and console-only errors; the Dockerfile CMD (line 29) is exactly the command an operator might copy into a systemd unit without the build step. Mitigations: both documented run paths build first (start:web runs vite build, package.json:10; Dockerfile RUN npm run build:web, line 20), and the fallback appears intentional for dev/tests (packages/web/test/server.test.mjs:10-12 pins staticRoot to source), so only off-path operation is affected. No security or fund-loss impact; Low (fact-kind doc/ops footgun) is correctly calibrated — fix is a one-line README note or a hard error/log when NODE_ENV=production and dist is missing.
